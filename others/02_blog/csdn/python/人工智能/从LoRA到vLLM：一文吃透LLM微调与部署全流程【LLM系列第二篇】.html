    从 LoRA 到 vLLM：一文吃透 LLM 微调与部署全流程【LLM 系列第二篇】

    上一篇我们从 GPT-1 一直讲到现代 LLM 的 5 大核心模块，属于"看懂 LLM 是什么"的阶段。但真到工作里，看懂模型只是 30 分——剩下 70 分是：怎么把开源大模型微调成自己业务能用的、再怎么把它高性能部署上线。

    这两件事是 LLM 工程化的绝对刚需，也是绝大多数 AI 岗位面试的高频考点。这篇打算用一条主线串起来：微调 → 部署，把从 SFT、LoRA、QLoRA 的原理，到 TRL/PEFT/BitsAndBytes 三库联动的实战，再到 vLLM 靠 PagedAttention 拉爆并发的秘密，全部讲清楚。

    看完你能拿捏这三件事：

    • 微调怎么选：什么时候用全参、什么时候用 LoRA、什么时候必须 QLoRA
    • 显存怎么算：给你一个 7B 模型，你能立刻估出微调要多少显存、瓶颈在哪
    • 部署怎么快：为什么 vLLM 比 Transformers 原生推理快 10 倍以上，PagedAttention 到底解决了什么问题

    一、引子：为什么每个 AI 工程师都要懂微调和部署？

    先问一个直击灵魂的问题：开源模型这么多，我为什么不能直接拿来用？

    拿一个真实场景：你要做一个"电商客服机器人"，用户问：我这个订单怎么退货？。

    如果你直接调 Qwen3-8B 的通用模型，答复会是这样：

    您好！退货流程通常分为以下几个步骤：
1. 登录您的账户……
2. 找到订单详情……
3. 提交退货申请……

    看起来没毛病对吧？但你老板会喷你——这是官方话术，不是我们家的话术。我们家客服要用"亲亲"开头、要引导用户找专属客服、要说清楚我们家 7 天无理由的具体规则。

    这就是微调（Fine-tuning）要解决的核心问题：让通用模型学会你的专属知识、风格和任务范式。

    一句话总结：预训练让模型有"通用智商"，微调让模型有"专业技能"。

    再退一步——就算你不做业务定制，只是自己玩玩，从 HuggingFace 拉个模型直接跑推理，你会发现：慢得离谱。一个 7B 模型单机跑并发，QPS 撑死几个，显存还蹭蹭往上涨。这时候你就需要 vLLM 这种专门的推理框架来救命。

    📖 参考仓库：微调代码参考

    二、核心概念：先把这 8 个词看明白

    老规矩，进入细节前先建立心智模型。这一节不堆公式，只用比喻：

    概念 一句话解释 生活比喻
    SFT 监督微调，用"问答对"教模型说话 学徒抄名家画作
    Base Model 只做过预训练、还不会聊天的"半成品" 只读过书没考过试的学生
    Instruct Model 已经指令微调过、会对话的"成品" 上过岗培训的应届生
    LoRA 只训练一小撮"外挂"权重，冻结主体 给西装配领带，不动西装本体
    QLoRA LoRA + 4-bit 量化，穷人版微调 把 A4 纸压缩成缩略图再改
    PagedAttention 借鉴操作系统分页机制管 KV Cache 硬盘分块存储避免"内存碎片"
    连续批处理 请求来了就塞、请求走了就腾位置 火锅店的流水座位，不排队等一桌
    FlashAttention 换个算法算注意力，少写显存少 IO 心算比列竖式快

    一句话总结 LLM 工程化的两大核心：微调是"让模型学会我们的活"，部署是"让模型高效干活"。

    三、微调整体流程：从数据到模型的完整闭环

    大模型微调看着神秘，其实和普通深度学习训练是一个模子——只是每一步都被"大"这个字放大了痛点。

    3.1 四步走

    1. 模型选择：Base Model 还是 Instruct Model？多大规模？
    2. 数据准备：从哪来？怎么清洗？什么格式？
    3. 微调训练：全参、LoRA 还是 QLoRA？怎么调超参？
    4. 模型验证：训练损失下降不够 → 看验证集 → 看样例推理 → 看真实业务效果

    3.2 模型选型：Base 还是 Instruct？

    大多数场景优先选 Instruct Model 作为微调起点。原因是它已经在 Base 之上做过指令微调 + 偏好对齐，具备指令理解和对话能力，数据准备和训练难度都低一档。

    场景 推荐 理由
    客服问答、内容生成、结构化输出 ✅ Instruct Model 已经会对话，只需注入领域知识
    需要深度改造模型行为的科研任务 ✅ Base Model 从"白纸"训起，可控性高
    团队有大量高质量领域数据 ✅ Base Model 数据量足够时能训得更彻底

    💡 判断标准：你只是想"改变模型说话方式或加点知识" → Instruct；你想"彻底重塑模型行为" → Base。

    3.3 数据格式：Alpaca vs ShareGPT vs OpenAI

    数据格式没有统一标准，但主流就三种。能记住这三种，看到任何数据集你都不慌：

    1️⃣ Alpaca 格式（指令式，单轮）

    源自斯坦福 Alpaca 项目，最简单：instruction / input / output 三个字段。

    {
  "instruction": "将以下英文翻译成中文",
  "input": "Large language models are transforming AI.",
  "output": "大语言模型正在改变人工智能。"
}

    2️⃣ ShareGPT 格式（对话式，多轮）

    来自 ChatGPT 早期用户分享的真实对话。用 conversations 数组 + from / value 字段。

    {
  "conversations": [
    {"from": "human", "value": "你能解释一下什么是机器学习吗？"},
    {"from": "gpt",   "value": "机器学习是人工智能的一个分支……"}
  ]
}

    3️⃣ OpenAI 格式（对话式，多轮，主流）

    OpenAI 微调 API 官方推荐。能同时支持单轮和多轮，且带 system 角色，最灵活。

    {
  "messages": [
    {"role": "system",    "content": "你是一名专业的营养师。"},
    {"role": "user",      "content": "下午容易疲惫，有什么健康零食？"},
    {"role": "assistant", "content": "推荐坚果、希腊酸奶、水果……"}
  ]
}

    💡 判断标准：新项目一律选 OpenAI 格式，兼容性最好、能覆盖单轮多轮、系统提示词位置最规范。

    不管选哪种，训练时都会通过 Chat Template（比如 ChatML）拼成一个统一字符串再喂给模型，你不用担心格式差异。

    一句话：数据决定上限，模型只是逼近上限。

    四、显存去哪了？—— LLM 微调的显存构成大剖析

    微调最大的敌人从来不是精度，是显存。你可能有一个 24GB 显存的 4090，看起来很牛，但一上 7B 模型全参微调直接 OOM。为什么？

    因为微调时显存占用远不止"模型参数"这一项。它是 5 个大头组成的：

    4.1 五大显存去向

    组成 公式 说明
    ① 模型参数 N_params × B_dtype 常驻显存，用于前向反向
    ② 梯度 N_params × B_dtype 和参数一样大
    ③ 优化器状态 2 × N_params × 4（AdamW，FP32） 通常是最大头
    ④ 激活值 b × s × h × l × B × k 前向中间结果，反向用
    ⑤ 其他 CUDA 上下文、通信缓冲、框架开销 小头，2~5 GB

    7B 模型 + AdamW + FP16 混合精度 举个例：

    ① 参数     : 7B × 2 bytes  = 14 GB
② 梯度     : 7B × 2 bytes  = 14 GB
③ 优化器   : 2 × 7B × 4    = 56 GB  ← 巨无霸
④ 激活值   : 视 batch/seq 而定
⑤ 其他     : ~ 3 GB
─────────────────────────────
合计（不含激活）≈ 90 GB

    运行结果：一张 80GB 的 A100 都装不下。全参微调 7B 至少要 2~4 张 80GB 卡。

    ⚠️ 关键洞察：优化器状态（AdamW 的一阶矩 \(m\)、二阶矩 \(v\)）必须用 FP32 保证数值稳定性，即使参数和梯度都能压到 FP16，优化器也压不下去——所以它成了显存瓶颈。

    这就是为什么 DeepSpeed ZeRO / LoRA / QLoRA 这些技术存在的根本原因：它们都在想办法搞掉那个 56GB 的优化器状态。

    4.2 单卡优化四板斧

    在你上分布式之前，先把单卡的能耗榨干。四个招式一起上，能救回不少显存：

    招式 核心思路 显存收益 代价
    梯度累积 大 batch 拆成多个 micro-batch 累加梯度 等效 batch 变大，显存不涨 无
    CPU 卸载 优化器状态/激活值卸载到 CPU 内存 中 PCIe 带宽变瓶颈
    梯度检查点 前向只存少量检查点，反向重算 大 训练时间增加 20~30%
    混合精度 FP16/BF16 加速计算，FP32 保底 参数梯度减半 需 Ampere+ 显卡

    💡 判断标准：优先开启混合精度和梯度检查点——性价比最高的两个开关。

    五、显存优化的重头戏：LoRA / QLoRA / ZeRO 三板斧

    单卡技巧能救几 GB，但要真正砸掉那个 56GB 优化器状态，得靠"结构性优化"。

    5.1 LoRA：只训练"外挂"，冻结主体 ⭐⭐⭐

    LoRA（Low-Rank Adaptation，低秩适应） 由微软 2021 年提出。它的洞察很朴素：微调时权重的"改变量" \(\Delta W\) 其实是低秩的——真正需要更新的自由度远比表面维度小。

    于是 LoRA 把 \(\Delta W\) 分解成两个小矩阵的乘积：

    \[ W = W_0 + \Delta W \approx W_0 + AB \]

    其中 \(A \in \mathbb{R}^{d \times r}\)，\(B \in \mathbb{R}^{r \times k}\)，秩 \(r \ll \min(d, k)\)，通常取 4、8、16。

    训练时冻结 \(W_0\)，只优化 \(A\) 和 \(B\)，推理时把 \(AB\) 合并回 \(W_0\)，零额外开销。

    参数压缩效果：一个 \(4096 \times 4096\) 的权重矩阵：

    全参更新     : 4096 × 4096         = 16,777,216  参数
LoRA (r=8)  : 4096 × 8 + 8 × 4096 =     65,536  参数

    结果：只训练 0.4% 的参数，效果却接近全参微调。

    LoRA 常插在哪？ Decoder-only 模型里最常插 q_proj 和 v_proj（Query 和 Value 对任务语义最敏感）。想稳一点可以四件套全上：q_proj / k_proj / v_proj / o_proj。

    💡 判断标准：单卡 24GB 想跑 7B → 上 LoRA；单卡 24GB 想跑 13B+ → 上 QLoRA。

    5.2 QLoRA：4-bit 量化 + LoRA，穷人的救命稻草

    QLoRA（Quantized LoRA） 是华盛顿大学 2023 年提出的 LoRA 增强版，靠三个技术把门槛压到消费级 GPU：

    1️⃣ 4-bit NormalFloat（NF4）量化：把 FP16 权重压到 4-bit，显存直接砍到 1/4。

    2️⃣ 双重量化（Double Quantization）：把量化用的缩放因子再量化一次。听着套娃，但确实能再省一点显存。

    3️⃣ 分页优化器（Paged Optimizer）：借鉴操作系统虚拟内存，让优化器状态能在 GPU/CPU 之间按需换页，避免 OOM。

    运行结果：QLoRA 让 7B 模型微调只需要 6GB 显存，一张 RTX 3060 就能跑。

    5.3 分布式：ZeRO 的分片策略 ⭐⭐

    当你有多张卡时，DeepSpeed 的 ZeRO（Zero Redundancy Optimizer） 是首选。核心思想一句话：别让每张 GPU 都保存完整模型状态，切片存、按需拉。

    三个递进阶段：

    阶段 切什么 显存节省 通信开销
    ZeRO-1 优化器状态 小 低
    ZeRO-2 优化器状态 + 梯度 中 中
    ZeRO-3 优化器状态 + 梯度 + 参数 大 高

    💡 判断标准：显存吃紧时提阶段（ZeRO-1 → 2 → 3），带宽吃紧时降阶段。ZeRO-3 通信量大，慢 NVLink 或 PCIe 用起来会痛。

    5.4 其他并行策略速览

    策略 切什么 适用场景
    数据并行 每卡完整模型 + 不同 batch 提升吞吐，模型能放进单卡
    流水线并行 按层纵向切 模型太大放不下，容忍气泡
    张量并行 单层内部矩阵横向切 模型太大且带宽足
    专家并行 MoE 的不同 Expert 切到不同卡 MoE 架构

    一句话：LoRA/QLoRA 是省显存的第一选择，ZeRO 是分布式的第一选择。

    六、上手实战：TRL + PEFT + BitsAndBytes 三库联动

    理论讲透了，来看真实代码。HuggingFace 生态里做微调的黄金三件套：

    • TRL：提供各种训练算法对应的 Trainer（SFTTrainer、DPOTrainer、GRPOTrainer……）
    • PEFT：参数高效微调库，LoRA/AdaLoRA 等就在这
    • BitsAndBytes：负责量化，让 QLoRA 成为可能

    三者协作关系一句话：TRL 负责训练流程 + PEFT 负责参数高效结构 + BitsAndBytes 负责量化底层。

    6.1 TRL 的核心两件套：SFTConfig + SFTTrainer

    SFTConfig 继承自 TrainingArguments，重要字段有这些：

    from trl import SFTConfig

config = SFTConfig(
    # 训练规模
    num_train_epochs=3,
    per_device_train_batch_size=8,
    gradient_accumulation_steps=1,

    # 优化关键
    bf16=True,                          # Ampere+ 强烈开启
    gradient_checkpointing=True,        # 时间换显存
    max_length=1500,

    # 保存与评估
    output_dir='./sft-output',
    save_strategy='steps', save_steps=100,
    eval_strategy='steps', eval_steps=100,
    load_best_model_at_end=True,

    # 学习率
    learning_rate=1e-5,                 # 全参 1e-5~5e-5，LoRA 1e-4~3e-4
    lr_scheduler_type='cosine',
    warmup_ratio=0.03,

    # SFT 灵魂开关 ⭐⭐⭐
    assistant_only_loss=True,           # 只算 assistant 部分的 loss
)

    🔥 assistant_only_loss —— SFT 训练最重要的一个开关

    默认会对整条输入（system + user + assistant）都算 loss，模型会"作弊"背诵固定 prompt。必须开启后只对 assistant 部分算 loss，才能和推理目标对齐。

    SFTTrainer 用法非常直接：

    from trl import SFTTrainer

trainer = SFTTrainer(
    model=model,
    processing_class=tokenizer,
    args=config,
    train_dataset=train_ds,
    eval_dataset=test_ds,
)
trainer.train()
trainer.save_model()

    6.2 加上 LoRA：PEFT 的三种接入方式

    PEFT 集成有三种方法，推荐第二种（把 peft_config 直接传给 Trainer）：

    from peft import LoraConfig
from trl import SFTTrainer

peft_config = LoraConfig(
    r=32,                                # 秩，越大表达能力越强、显存越多
    lora_alpha=16,                       # 缩放系数
    lora_dropout=0.05,                   # 防过拟合
    bias='none',
    task_type='CAUSAL_LM',
    target_modules=['q_proj', 'v_proj', 'k_proj', 'o_proj'],
)

trainer = SFTTrainer(
    model='Qwen/Qwen2.5-7B',
    args=training_args,
    train_dataset=dataset,
    peft_config=peft_config,             # 一行接入
)
trainer.train()

    💡 判断标准：PEFT 学习率要比全参高 ~10 倍（1e-4 起），因为可训练参数少，需要更大的更新步长。

    6.3 上量化：BitsAndBytes 让 QLoRA 一键启用

    三步走：

    # 1️⃣ 构造量化配置
from transformers import BitsAndBytesConfig
import torch

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,                    # 启用 4-bit
    bnb_4bit_use_double_quant=True,       # 双重量化
    bnb_4bit_quant_type='nf4',            # NF4 量化
    bnb_4bit_compute_dtype=torch.bfloat16,# 计算精度
)

# 2️⃣ 用量化配置加载模型
from transformers import AutoModelForCausalLM

model = AutoModelForCausalLM.from_pretrained(
    'Qwen/Qwen2.5-7B',
    quantization_config=bnb_config,
    device_map='auto',
)

# 3️⃣ 挂上 LoRA，把 SFTTrainer 拉起来
trainer = SFTTrainer(
    model=model,
    args=training_args,
    train_dataset=dataset,
    peft_config=peft_config,              # LoRA 配置
)
trainer.train()

    运行结果：Qwen2.5-7B 全参微调要 90GB+，QLoRA 之后只要 6GB 左右——14 倍显存压缩。

    6.4 加速终极招：Unsloth

    如果你觉得 QLoRA 还不够快，还有 Unsloth——它把 attention、rmsnorm、rope 等关键算子重写成手工优化 kernel，速度比原生 HuggingFace 快 2~5 倍，显存再省 50%。API 也和 TRL 兼容。

    一句话：TRL 出招式、PEFT 出结构、BitsAndBytes 出量化、Unsloth 出加速——四件套下来，个人开发者也能玩转 30B+ 模型。

    七、部署上线：vLLM 与推理框架的四个大招

    模型微调好了，怎么部署？

    答案是：用推理框架，不要直接用 Transformers 原生 pipeline。原生 .generate() 只能算个"能跑"，一上并发就露怯。

    7.1 为什么需要专用推理框架？

    推理场景和训练场景完全不同——训练是"离线批处理"，推理是"在线高并发"。要解决三个核心问题：

    1. 调度问题：多个请求同时来，怎么高效排队和切换？
    2. 显存问题：长上下文 + 高并发下，KV Cache 怎么管才不炸？
    3. 性能问题：如何最大化 GPU 利用率、降低时延？

    vLLM 是这个领域的绝对头部——底层基于 PyTorch，为大规模并发场景重构了整套推理流水线。

    7.2 vLLM 极简部署

    # 安装（Linux only，Windows 用 WSL2）
uv pip install vllm

# 前台起服务
vllm serve ./Qwen3-8B --served-model-name Qwen3-8B --max-model-len 32K

# 后台起服务
nohup vllm serve /path/to/Qwen3-8B --served-model-name Qwen3-8B \
    --max-model-len 32K > vllm.log 2>&1 &

    启动后 vLLM 会开一个和 OpenAI API 完全兼容的 HTTP 服务，你的应用改个 base_url 就能无缝切换。

    ⚠️ WSL2 部署踩坑集锦

    • RuntimeError: UVA is not available → 环境变量 VLLM_WSL2_ENABLE_PIN_MEMORY=1
    • 找不到 gcc → sudo apt install build-essential
    • nvcc 相关错误 → 必须装完整的 CUDA Toolkit（PyTorch 自带的 runtime 不够，vLLM 需要真正的 nvcc）

    7.3 大招 1：PagedAttention —— vLLM 的看家本领 ⭐⭐⭐

    传统 KV Cache 有一个致命问题：每个请求分配的显存必须连续，用完之后归还，长期运行下会产生大量"显存碎片"——总空间够，但没有一段连续空间可用。

    熟悉操作系统的读者应该秒懂——这不就是"内存碎片"吗？OS 是怎么解决的？分页机制。

    PagedAttention 直接把这套思想搬到了 KV Cache 上：

    • 预先申请一个大的 KV Cache 显存池
    • 划分成固定大小的 block（比如 16 个 token 一块）
    • 每个请求的 KV 序列在物理上不连续，靠一张 block table 索引
    • Block Manager 统一负责分配和回收

    额外福利：前缀共享。当多个请求有相同的 system prompt 或前缀时，它们的 KV 完全一样——PagedAttention 让这些请求共享同一批 block，显存进一步省下来：

    7.4 大招 2：连续批处理（Continuous Batching）

    传统批处理是"静态"的——凑够一个 batch 才开始跑，中途请求完成了也得等最长的那个跑完。这就像火锅店必须凑够 4 个人才能上桌。

    连续批处理换了个思路：每一步都重新组织 batch。新请求随时插入、完成请求立刻退出，GPU 永远保持满载。

    调度器的五个组件：

    组件 职责
    Waiting Queue 存放尚未开始执行的新请求
    Running Queue 存放已完成 Prefill、正在 Decode 的请求
    KV Cache Pool 所有请求的 KV，由 Block Manager 管理
    Scheduler 每轮决定跑哪些请求
    Continuous Batching 每轮重组 Batch，动态增删请求

    运行结果：吞吐量相比静态批处理提升 5~10 倍，GPU 利用率从 30% 拉到 80%+。

    7.5 大招 3：FlashAttention —— 让 Attention 少读几次显存

    标准 Attention 会把中间的 \(QK^T\)、softmax 结果全部存到显存里。FlashAttention 换一种算法：分块（tiling）+ 在线 softmax，把这些中间结果尽量留在 SRAM（GPU 片上高速缓存）里，避免频繁读写 HBM（显存）。

    结果就是：同样的注意力计算，速度快 2~3 倍，长上下文场景显存省一半。你只需要在加载模型时传 attn_implementation='flash_attention_2'，享受免费加速。

    7.6 大招 4：CUDA Graph + Tensor Parallel

    CUDA Graph：在 decode 阶段，每步执行的 kernel 形状高度相似。传统模式下 CPU 得逐个提交 kernel launch，开销不小。CUDA Graph 把这些 kernel 打包成一张图，重放整张图，砍掉 CPU 调度开销。

    Tensor Parallel：单卡放不下时，把每层的矩阵计算按输出维度切到多卡上，vLLM 一个 --tensor-parallel-size 4 就能开。

    一句话：vLLM = PagedAttention + 连续批处理 + FlashAttention + CUDA Graph + Tensor Parallel——一套组合拳打下来，比原生 Transformers 快 10~24 倍。

    八、避坑指南：8 条微调 & 部署铁律

    铁律 1：assistant_only_loss 必开

    # ❌ 默认对整条输入都算 loss，模型学会"背诵 prompt"
SFTConfig(...)

# ✅ 只对 assistant 部分算 loss
SFTConfig(..., assistant_only_loss=True)

    💡 判断标准：对话式数据、多轮 SFT，一律开启。

    铁律 2：LoRA 学习率要拉高 10 倍

    # ❌ 用全参微调的学习率跑 LoRA，模型学不动
learning_rate = 1e-5

# ✅ LoRA 至少 1e-4 起步
learning_rate = 2e-4

    💡 判断标准：全参 1e-5~5e-5，LoRA 1e-4~3e-4。可训练参数越少，学习率越大。

    铁律 3：eval_strategy 和 save_strategy 保持一致

    # ❌ 一个按 epoch，一个按 steps，load_best_model_at_end 会崩
eval_strategy = 'epoch'
save_strategy = 'steps'
load_best_model_at_end = True

# ✅ 保持一致
eval_strategy = 'steps'
save_strategy = 'steps'

    💡 判断标准：开启 load_best_model_at_end 时，两个 strategy 必须相同。

    铁律 4：显存 OOM 排查按顺序来

    1. 减小 per_device_train_batch_size （8 → 4 → 2 → 1）
2. 开启 gradient_checkpointing=True
3. 开启 bf16=True（Ampere+）或 fp16=True
4. 上 LoRA（peft_config）
5. 还不行，上 QLoRA（BitsAndBytesConfig 4-bit）
6. 还不行，上多卡 + ZeRO

    💡 判断标准：从便宜的招式开始试，别一上来就动结构。

    铁律 5：LoRA 别乱插层

    # ❌ 全线性层无脑塞 LoRA，参数量爆炸
target_modules = 'all-linear'    # 谨慎使用

# ✅ 主流做法：注意力四件套
target_modules = ['q_proj', 'k_proj', 'v_proj', 'o_proj']

    💡 判断标准：先插 q_proj/v_proj，效果不够再加 k_proj/o_proj，最后才考虑 FFN 层。

    铁律 6：数据质量 > 数据数量

    # ❌ 塞 100 万条噪声数据
train_ds = 1_000_000 条从网上爬的对话

# ✅ 精挑 1 万条业务真实数据 + 人工审核
train_ds = 10_000 条清洗过的高质量样本

    💡 判断标准：1 万条精品数据的效果 > 100 万条水货数据——LLaMA 论文和 LIMA 论文早就证明过。

    铁律 7：vLLM 部署一定要限 max-model-len

    # ❌ 不设 max-model-len，vLLM 按训练最大值预留 KV Cache（可能 128K）
vllm serve ./Qwen3-8B

# ✅ 按业务实际需求设，把显存留给并发
vllm serve ./Qwen3-8B --max-model-len 8192

    💡 判断标准：根据业务真实上下文长度设置——设小了截断，设大了浪费显存降并发。

    铁律 8：模型保存分清楚"完整模型"和"adapter"

    # ❌ LoRA 微调后直接分享，别人拿到只是 adapter 权重
trainer.save_model('./output')    # 只保存了 adapter

# ✅ 想分享完整模型，先 merge
model = model.merge_and_unload()  # 把 LoRA 合并回主体
model.save_pretrained('./full-model')

    💡 判断标准：分享推理服务 → merge 后保存完整模型；自己继续训 → 保存 adapter 即可。

    九、总结：3 张速查表 + 1 句口诀

    9.1 微调方案选型速查表

    场景 显存 推荐方案 学习率
    单卡 80GB，7B 模型 90 GB 需求 全参微调 + ZeRO-2 (双卡) 1e-5
    单卡 24GB，7B 模型 20 GB 需求 LoRA 2e-4
    单卡 8GB，7B 模型 6 GB 需求 QLoRA (4bit) 2e-4
    消费级 3060，13B+ 模型 8~10 GB QLoRA + Unsloth 2e-4
    有多张 80GB 卡，70B 模型 分布式 ZeRO-3 + Tensor Parallel 1e-5

    9.2 SFTConfig 关键参数速查表

    参数 推荐值 说明
    bf16 True Ampere+ 一律开
    gradient_checkpointing True 时间换显存
    assistant_only_loss True 对话式 SFT 必开
    learning_rate 全参 1e-5，LoRA 2e-4 PEFT ~10 倍
    warmup_ratio 0.03~0.1 稳定训练
    lr_scheduler_type cosine 收敛更平滑
    save_total_limit 2~3 别把硬盘塞爆

    9.3 vLLM 部署优化速查表

    优化 参数 / 配置 收益
    PagedAttention 默认开启 显存碎片 → 0
    连续批处理 默认开启 吞吐量 × 5~10
    FlashAttention 2 --attention-backend FLASH_ATTN 速度 × 2~3
    Tensor Parallel --tensor-parallel-size N 单卡放不下时
    CUDA Graph 默认开启 Decode 更稳定
    前缀缓存 --enable-prefix-caching 相同 system prompt 提速

    9.4 一句话口诀

    微调三板斧：LoRA 压参数 / QLoRA 压显存 / ZeRO 压分布式；部署四大招：PagedAttention 管 KV / 连续批处理管调度 / FlashAttention 管算子 / Tensor Parallel 管切分。

    更精炼一版："能 LoRA 就别全参，能 QLoRA 就别 LoRA，能 vLLM 就别 pipeline"——降门槛、提性能，永远这三步。
