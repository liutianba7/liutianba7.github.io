    FastAPI 数据库篇：SQLAlchemy 2.0 异步 ORM 从概念到 CRUD 实战

    1. 引子：为什么很多人学 SQLAlchemy 会卡住？

    写 FastAPI 项目，绕不开数据库。可 SQLAlchemy 这东西，新手最容易陷入一个尴尬状态：代码能抄着写，但脑子里没有地图。

    比如这些问题，看起来都是小问题，实际都指向同一个核心：你没有真正理解 SQLAlchemy 的运行机制。

    • engine、connection、session 到底谁负责连数据库？
    • 为什么安装了 sqlalchemy，还要再装 aiomysql、asyncpg、aiosqlite？
    • session.add(obj) 之后数据库为什么还没数据？
    • commit() 之后为什么有时还要 refresh()？
    • select(Book) 为什么不能直接拿到结果，还要 execute()、scalars()、all()？
    • FastAPI 里为什么要用 yield 依赖管理数据库 Session？

    SQLAlchemy 不是简单的“Python 版 SQL”，它更像一个数据库操作系统：

    Engine 管连接，Session 管事务，Model 管对象，Driver 负责真正和数据库通信。

    上一篇我们聊了 FastAPI 的中间件、安全认证、后台任务和工程化，这一篇进入后端服务最核心的部分：数据库访问层。

    2. 核心概念解析

    2.1 SQLAlchemy 到底是什么？

    很多人把 SQLAlchemy 简单理解成 ORM，这其实只说对了一半。

    SQLAlchemy 实际上由两层组成：

    层次 名称 作用 类比
    Core SQL 表达式语言 + 执行系统 用 Python 代码构造 SQL、执行 SQL 高级 SQL 拼装器
    ORM 对象关系映射 把数据库表映射成 Python 类，把行映射成对象 表变类，行变对象

    也就是说：

    # Core 风格：构造 SQL 表达式
stmt = select(users.c.name).where(users.c.age > 18)

# ORM 风格：操作 Python 类
stmt = select(User).where(User.age > 18)

    SQLAlchemy ORM 是建立在 Core 之上的。ORM 负责对象映射，Core 负责 SQL 构造和执行。

    这也是为什么 SQLAlchemy 2.0 推荐你用这种写法：

    stmt = select(Book).where(Book.id == 1)
result = await session.execute(stmt)
book = result.scalar_one_or_none()

    而不是老式的：

    # 老式写法，SQLAlchemy 2.0 中不推荐作为主流写法
book = session.query(Book).filter(Book.id == 1).first()

    SQLAlchemy 2.0 的核心思想是：统一 Core 和 ORM 的查询风格。

    2.2 为什么 SQLAlchemy 还需要数据库驱动？

    SQLAlchemy 本身不会直接和数据库服务器“打电话”。真正负责网络通信的是数据库驱动。

    可以把整个链路理解成四层：

    你的 Python 代码
    ↓
SQLAlchemy ORM / Core
    ↓
数据库驱动：aiomysql / asyncpg / aiosqlite
    ↓
数据库服务器：MySQL / PostgreSQL / SQLite

    【建议配图：SQLAlchemy 四层架构图】 展示 Python 代码 → SQLAlchemy ORM/Core → DBAPI Driver → Database Server 的调用链。

    AI 绘图提示词：

    A clean technical architecture diagram showing the four-layer architecture of SQLAlchemy. Top layer: Python application code. Second layer: SQLAlchemy ORM and Core. Third layer: DBAPI drivers such as aiomysql, asyncpg, aiosqlite. Bottom layer: database servers MySQL, PostgreSQL, SQLite. Use blue and white color scheme, professional software architecture style, clear arrows from top to bottom.

    用生活里的比喻解释：

    • SQLAlchemy 像“语法老师”，负责把你的 Python 表达式组织成 SQL
    • 数据库驱动像“电话员”，负责通过 TCP 协议把 SQL 发给数据库
    • 数据库服务器像“办事大厅”，真正执行查询、插入、更新、删除

    不同数据库需要不同驱动：

    数据库 同步驱动 异步驱动 说明
    MySQL pymysql aiomysql FastAPI 异步项目常用 aiomysql
    PostgreSQL psycopg2 asyncpg asyncpg 性能很好，生产常用
    SQLite sqlite3 aiosqlite 学习和本地开发最方便

    因此，下面这条安装命令不是重复，而是分工明确：

    pip install "sqlalchemy[asyncio]" aiosqlite
    • sqlalchemy[asyncio]：让 SQLAlchemy 支持异步接口
    • aiosqlite：真正负责和 SQLite 数据库通信

    如果换成 MySQL：

    pip install "sqlalchemy[asyncio]" aiomysql

    如果换成 PostgreSQL：

    pip install "sqlalchemy[asyncio]" asyncpg

    2.3 Engine、Connection、Session 的关系

    这是 SQLAlchemy 最重要的一组概念。

    概念 一句话 类比
    Engine 数据库连接总入口，内部维护连接池 出租车公司
    Connection 真正的一条数据库连接 一辆出租车
    Session ORM 工作区，管理事务和对象状态 一次出行的行程单

    更直观地说：

    Engine（连接工厂 + 连接池）
  │
  ├── 创建 / 管理 Connection（真实数据库连接）
  │
  └── 被 Session 使用
          │
          ├── 管理事务 commit / rollback
          ├── 管理 ORM 对象状态
          └── 需要执行 SQL 时，向 Engine 借 Connection

    【建议配图：Engine / Session / Connection 关系图】 展示 Engine 维护连接池，Session 是工作区，Session 执行 SQL 时从 Engine 借 Connection，事务结束后 Connection 回池。

    AI 绘图提示词：

    A professional diagram explaining SQLAlchemy Engine, Session, and Connection relationship. Engine contains a Connection Pool with multiple database connections. Session is shown as a workspace that borrows a Connection from Engine when executing SQL, manages ORM objects and transaction, then returns the Connection to the pool after commit or rollback. Clean modern style, blue and gray theme, arrows and labels.

    注意一个关键点：

    Session 不是数据库连接。Session 是工作区。

    Session 里面可以有很多 ORM 对象，它会记录这些对象哪些新增了、哪些修改了、哪些删除了。只有当你 commit() 时，这些变化才会真正同步到数据库。

    2.4 Session 为什么像 Git 暂存区？

    如果你熟悉 Git，可以这样理解 SQLAlchemy Session：

    Git 操作 SQLAlchemy Session 含义
    git add file session.add(obj) 标记对象待保存
    git diff 脏追踪 记录哪些对象被改了
    git commit session.commit() 真正提交到数据库
    git reset session.rollback() 撤销本次事务变更

    比如：

    book = Book(title="Python 入门", price=99.0)
session.add(book)

# 到这里，数据库里还没有这本书
# add() 只是把对象放进 Session 工作区

await session.commit()

# commit() 之后，INSERT SQL 才真正执行

    这就是很多新手困惑的地方：

    session.add(book)

    不等于写入数据库，它只是告诉 Session：

    “这个对象等会儿提交时帮我保存一下。”

    3. 核心体系：SQLAlchemy 2.0 异步 ORM 工作流

    接下来按照真实项目开发顺序，把 SQLAlchemy 的核心体系串起来。

    3.1 安装与连接 URL

    以 SQLite 为例，学习成本最低：

    pip install "sqlalchemy[asyncio]" aiosqlite

    如果使用 uv：

    uv add "sqlalchemy[asyncio]" aiosqlite

    数据库连接 URL 长这样：

    DATABASE_URL = "sqlite+aiosqlite:///./books.db"

    如果是 MySQL：

    DATABASE_URL = "mysql+aiomysql://root:password@localhost:3306/mydb?charset=utf8mb4"

    拆开看：

    mysql+aiomysql://root:password@localhost:3306/mydb?charset=utf8mb4
  │       │          │       │        │       │
  │       │          │       │        │       └── 字符集
  │       │          │       │        └────────── 数据库名
  │       │          │       └─────────────────── 端口
  │       │          └─────────────────────────── 主机
  │       └────────────────────────────────────── 驱动
  └────────────────────────────────────────────── 数据库类型

    连接 URL 的格式决定了 SQLAlchemy 使用哪个数据库、哪个驱动、连到哪里。

    3.2 创建 Engine 与 Session 工厂

    新建 database.py：

    from typing import AsyncGenerator

from sqlalchemy.ext.asyncio import (
    AsyncSession,
    async_sessionmaker,
    create_async_engine,
)
from sqlalchemy.orm import DeclarativeBase

# SQLite 异步连接 URL
DATABASE_URL = "sqlite+aiosqlite:///./books.db"

# 创建异步 Engine
# echo=True 会打印 SQL，开发环境很有用；生产环境建议 False
engine = create_async_engine(
    DATABASE_URL,
    echo=True,
)

# 创建异步 Session 工厂
# expire_on_commit=False 非常重要，后面避坑会详细讲
AsyncSessionLocal = async_sessionmaker(
    bind=engine,
    class_=AsyncSession,
    expire_on_commit=False,
    autoflush=False,
    autocommit=False,
)

# ORM 模型基类
class Base(DeclarativeBase):
    pass

# FastAPI 依赖：每个请求创建一个 Session
async def get_db() -> AsyncGenerator[AsyncSession, None]:
    async with AsyncSessionLocal() as session:
        try:
            yield session
            # 如果接口没有手动提交，这里统一提交
            await session.commit()
        except Exception:
            # 出错自动回滚，避免脏数据写入
            await session.rollback()
            raise

    这段代码是 FastAPI + SQLAlchemy 异步项目的核心模板。

    执行过程如下：

    请求进来
  ↓
get_db 创建 AsyncSession
  ↓
Session 注入到接口函数
  ↓
接口执行查询 / 新增 / 更新 / 删除
  ↓
成功 commit，异常 rollback
  ↓
Session 关闭，Connection 归还连接池

    3.3 定义模型：Mapped + mapped_column

    SQLAlchemy 2.0 推荐使用 Mapped[T] + mapped_column() 定义模型。

    新建 models.py：

    from datetime import datetime
from typing import Annotated

from sqlalchemy import String, Text, func
from sqlalchemy.orm import Mapped, mapped_column

from database import Base

# 可复用字段配置：主键 ID
int_pk = Annotated[int, mapped_column(primary_key=True, autoincrement=True)]

# 可复用字段配置：短字符串
str_100 = Annotated[str, mapped_column(String(100))]

class Book(Base):
    """图书表模型"""
    __tablename__ = "books"

    # 主键：Mapped[int_pk] 会自动应用 primary_key=True
    id: Mapped[int_pk]

    # 书名：String(100)，不可为空
    title: Mapped[str_100]

    # 作者：String(100)，不可为空
    author: Mapped[str_100]

    # 价格：float 会自动映射为 Float 类型
    price: Mapped[float]

    # 描述：str | None 表示 Python 层可为 None，数据库层默认 nullable=True
    description: Mapped[str | None] = mapped_column(Text, default=None)

    # 创建时间：由数据库服务器生成默认值
    created_at: Mapped[datetime] = mapped_column(server_default=func.now())

    重点理解三个概念：

    写法 含义
    Mapped[int] 声明这是一个 ORM 映射字段
    mapped_column() 配置数据库列，比如主键、长度、默认值
    str | None 自动推导数据库列可以为 NULL

    3.4 Pydantic Schema：请求模型和响应模型分离

    新建 schemas.py：

    from datetime import datetime
from pydantic import BaseModel, Field

class BookCreate(BaseModel):
    """创建图书的请求体"""
    title: str = Field(..., min_length=1, max_length=100, description="书名")
    author: str = Field(..., min_length=1, max_length=100, description="作者")
    price: float = Field(..., gt=0, description="价格必须大于 0")
    description: str | None = Field(None, description="图书描述")

class BookUpdate(BaseModel):
    """更新图书的请求体：所有字段可选，支持部分更新"""
    title: str | None = Field(None, min_length=1, max_length=100)
    author: str | None = Field(None, min_length=1, max_length=100)
    price: float | None = Field(None, gt=0)
    description: str | None = None

class BookOut(BaseModel):
    """响应模型"""
    id: int
    title: str
    author: str
    price: float
    description: str | None
    created_at: datetime

    # Pydantic v2：允许从 ORM 对象读取字段
    model_config = {
        "from_attributes": True
    }

    为什么要分开？

    • BookCreate：客户端创建时需要传什么
    • BookUpdate：客户端更新时允许传什么
    • BookOut：服务端响应时暴露什么

    数据库模型负责“存储结构”，Pydantic 模型负责“接口结构”。不要混用。

    3.5 查询体系：select → execute → Result

    SQLAlchemy 2.0 查询分三步：

    stmt = select(Book)                 # 1. 构造 SQL，不执行
result = await session.execute(stmt) # 2. 执行 SQL，返回 Result
books = result.scalars().all()       # 3. 提取 ORM 对象列表

    很多人卡在 Result 上。

    session.execute() 返回的不是 Book，而是一个结果包装器。你要根据查询内容决定怎么取值：

    方法 返回值 适用场景
    scalars().all() list[Book] 查询多个 ORM 对象
    scalars().first() Book | None 取第一条，可为空
    scalar_one() Book 必须刚好一条，否则报错
    scalar_one_or_none() Book | None 最多一条，多条报错
    scalar() 单个值 count、sum、avg 这类聚合查询
    all() list[Row] 多列查询、JOIN 查询

    举例：

    from sqlalchemy import select, func

# 查询多个 ORM 对象
result = await session.execute(select(Book))
books = result.scalars().all()

# 查询单个 ORM 对象
result = await session.execute(select(Book).where(Book.id == 1))
book = result.scalar_one_or_none()

# 查询总数：这里返回的是单个数字，所以用 scalar()
result = await session.execute(select(func.count(Book.id)))
total = result.scalar()

    4. 实战代码演示：图书管理 API

    下面写一个完整可运行的 FastAPI + SQLAlchemy 2.0 异步 ORM 示例。

    功能包括：

    • 初始化数据库
    • 创建图书
    • 分页查询图书列表
    • 查询图书详情
    • 部分更新图书
    • 删除图书

    4.1 项目结构

    book_api/
├── main.py
├── database.py
├── models.py
└── schemas.py

    为了方便阅读，上面几个模块的代码已经在前面拆开解释了，下面给出完整 main.py。

    4.2 完整 main.py

    from contextlib import asynccontextmanager
from typing import Annotated

from fastapi import Depends, FastAPI, HTTPException, Query, status
from sqlalchemy import delete, func, select
from sqlalchemy.ext.asyncio import AsyncSession

from database import Base, engine, get_db
from models import Book
from schemas import BookCreate, BookOut, BookUpdate


@asynccontextmanager
async def lifespan(app: FastAPI):
    """应用生命周期：启动时创建数据库表"""
    # 学习阶段可以自动建表；生产环境建议使用 Alembic
    async with engine.begin() as conn:
        await conn.run_sync(Base.metadata.create_all)
    yield


app = FastAPI(title="Book API", lifespan=lifespan)


@app.post(
    "/books/",
    response_model=BookOut,
    status_code=status.HTTP_201_CREATED,
)
async def create_book(
    book_in: BookCreate,
    db: Annotated[AsyncSession, Depends(get_db)],
):
    """创建图书"""
    # 1. 将 Pydantic 请求模型转换为 ORM 对象
    book = Book(**book_in.model_dump())

    # 2. 加入 Session 工作区，此时数据库还没有数据
    db.add(book)

    # 3. 提交事务，真正执行 INSERT
    await db.commit()

    # 4. 刷新对象，拿到数据库生成的 id、created_at 等字段
    await db.refresh(book)

    return book


@app.get("/books/", response_model=dict)
async def list_books(
    db: Annotated[AsyncSession, Depends(get_db)],
    page: Annotated[int, Query(ge=1, description="页码，从 1 开始")] = 1,
    size: Annotated[int, Query(ge=1, le=100, description="每页数量")] = 10,
    keyword: Annotated[str | None, Query(description="按书名模糊搜索")] = None,
):
    """分页查询图书列表"""
    # 基础查询
    stmt = select(Book)
    count_stmt = select(func.count(Book.id))

    # 可选搜索条件
    if keyword:
        stmt = stmt.where(Book.title.contains(keyword))
        count_stmt = count_stmt.where(Book.title.contains(keyword))

    # 查询总数
    total = (await db.execute(count_stmt)).scalar() or 0

    # 分页 + 排序
    stmt = (
        stmt
        .order_by(Book.id.desc())
        .offset((page - 1) * size)
        .limit(size)
    )

    result = await db.execute(stmt)
    books = result.scalars().all()

    return {
        "total": total,
        "page": page,
        "size": size,
        "items": [BookOut.model_validate(book).model_dump() for book in books],
    }


@app.get("/books/{book_id}", response_model=BookOut)
async def get_book(
    book_id: int,
    db: Annotated[AsyncSession, Depends(get_db)],
):
    """查询单本图书"""
    stmt = select(Book).where(Book.id == book_id)
    result = await db.execute(stmt)
    book = result.scalar_one_or_none()

    if book is None:
        raise HTTPException(status_code=404, detail="图书不存在")

    return book


@app.patch("/books/{book_id}", response_model=BookOut)
async def update_book(
    book_id: int,
    book_in: BookUpdate,
    db: Annotated[AsyncSession, Depends(get_db)],
):
    """部分更新图书"""
    # 1. 先查后改，方便做存在性校验和业务逻辑
    stmt = select(Book).where(Book.id == book_id)
    result = await db.execute(stmt)
    book = result.scalar_one_or_none()

    if book is None:
        raise HTTPException(status_code=404, detail="图书不存在")

    # 2. exclude_unset=True 只取用户实际传入的字段
    update_data = book_in.model_dump(exclude_unset=True)

    # 3. 动态更新字段
    for field, value in update_data.items():
        setattr(book, field, value)

    # 4. 提交事务，SQLAlchemy 会自动追踪哪些字段变了
    await db.commit()
    await db.refresh(book)

    return book


@app.delete("/books/{book_id}")
async def delete_book(
    book_id: int,
    db: Annotated[AsyncSession, Depends(get_db)],
):
    """删除图书"""
    # 方式一：先查后删，适合需要业务校验的场景
    stmt = select(Book).where(Book.id == book_id)
    result = await db.execute(stmt)
    book = result.scalar_one_or_none()

    if book is None:
        raise HTTPException(status_code=404, detail="图书不存在")

    await db.delete(book)
    await db.commit()

    return {"message": "删除成功"}


@app.delete("/books/batch/by-author/{author}")
async def delete_books_by_author(
    author: str,
    db: Annotated[AsyncSession, Depends(get_db)],
):
    """批量删除某个作者的图书"""
    stmt = delete(Book).where(Book.author == author)
    result = await db.execute(stmt)
    await db.commit()

    return {"deleted": result.rowcount}

    5. 避坑指南 / 最佳实践

    5.1 add() 不等于写入数据库

    book = Book(title="Python", author="A", price=99)
db.add(book)

# 此时数据库没有任何变化
# 只是 Session 记录：这个对象待插入

await db.commit()

# commit 后才真正执行 INSERT

    记住：Session 是工作区，commit 才是提交。

    5.2 什么时候需要 refresh()？

    refresh() 会重新从数据库加载对象，主要用于拿数据库生成的字段：

    • 自增主键 ID
    • server_default 生成的时间
    • 数据库触发器生成的字段
    • 数据库层面自动修改的字段
    db.add(book)
await db.commit()
await db.refresh(book)

print(book.id)          # 自增 ID
print(book.created_at)  # 数据库生成的创建时间

    如果所有字段都是 Python 自己设置的，不依赖数据库生成，可以不 refresh。

    5.3 expire_on_commit=False 为什么重要？

    SQLAlchemy 默认在 commit() 后会让对象属性“过期”。下次访问属性时，它会尝试重新查数据库。

    同步环境中问题不大，但异步环境中，隐式 IO 很危险：

    await db.commit()
print(book.title)  # 如果属性过期，可能触发隐式数据库查询

    异步代码应该显式 await，不应该偷偷发 IO。所以异步项目一般设置：

    AsyncSessionLocal = async_sessionmaker(
    engine,
    expire_on_commit=False,
)

    这样 commit 后对象属性仍然可用。

    5.4 scalars() 和 scalar() 不要混

    这两个名字很像，但用途不同。

    方法 用途 示例
    scalars().all() 多个 ORM 对象 select(Book)
    scalar_one_or_none() 一个 ORM 对象 select(Book).where(Book.id == 1)
    scalar() 单个值 select(func.count(Book.id))

    错误示例：

    # 查询 count，却用了 scalars().all()，结果会很奇怪
result = await db.execute(select(func.count(Book.id)))
total = result.scalars().all()  # 返回类似 [10]

    正确写法：

    result = await db.execute(select(func.count(Book.id)))
total = result.scalar()

    5.5 更新时一定要用 exclude_unset=True

    部分更新最容易踩坑：

    class BookUpdate(BaseModel):
    title: str | None = None
    price: float | None = None

    如果用户只传了：

    {"price": 88}

    普通 model_dump() 会得到：

    {"title": None, "price": 88}

    如果直接更新，title 会被错误地改成 None。

    正确写法：

    update_data = book_in.model_dump(exclude_unset=True)
# 结果：{"price": 88}

    5.6 生产环境不要自动建表

    学习阶段可以：

    await conn.run_sync(Base.metadata.create_all)

    但生产环境不要这么干。

    原因：

    • 自动建表不会处理字段修改
    • 不会安全地删除字段
    • 不会管理索引变更
    • 没有迁移历史
    • 多人协作容易失控

    生产环境建议使用 Alembic：

    alembic init alembic
alembic revision --autogenerate -m "create books table"
alembic upgrade head

    5.7 原生 SQL 必须参数绑定

    错误写法：

    # 存在 SQL 注入风险
sql = text(f"SELECT * FROM books WHERE author = '{author}'")

    正确写法：

    from sqlalchemy import text

sql = text("SELECT * FROM books WHERE author = :author")
result = await db.execute(sql, {"author": author})

    参数绑定让驱动负责转义和传参，避免 SQL 注入。

    6. 总结

    SQLAlchemy 2.0 异步 ORM 的学习重点，不是背 API，而是搞清楚每个组件的职责。

    概念 作用 一句话
    Engine 管理连接池 数据库连接总入口
    Driver 与数据库通信 真正发送 SQL 的人
    Connection 真实连接 一条数据库连接
    Session ORM 工作区 管事务、对象状态、缓存
    Model 表映射 Python 类对应数据库表
    Mapped 字段声明 告诉 ORM 这是映射属性
    mapped_column 列配置 设置主键、类型、默认值
    select() 构造查询 只构造，不执行
    execute() 执行 SQL 真正访问数据库
    Result 查询结果包装器 需要 scalars() / scalar() 提取
    commit() 提交事务 真正写入数据库
    rollback() 回滚事务 异常时撤销变更
    refresh() 重新加载对象 获取数据库生成字段
    Alembic 数据库迁移 生产环境管理表结构变化

    把这张表记住，SQLAlchemy 的主线就清晰了：

    Model 定义表结构
  ↓
Session 创建工作区
  ↓
select / add / update / delete 构造操作
  ↓
execute / commit 真正访问数据库
  ↓
Result / ORM 对象返回给业务层

    数据库篇到这里，FastAPI 后端开发的三块核心就串起来了：

    • 基础篇：路由、参数、响应、异常、依赖注入
    • 高级篇：中间件、安全认证、后台任务、WebSocket、工程化
    • 数据库篇：SQLAlchemy 2.0 异步 ORM、CRUD、事务与迁移
