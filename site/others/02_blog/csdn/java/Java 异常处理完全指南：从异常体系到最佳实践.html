    Java 异常处理完全指南：从异常体系到最佳实践

    一、从一个"血泪"故事开始

    先看一段代码，猜猜有什么问题：

    public void readConfig(String path) {
    FileInputStream fis = null;
    try {
        fis = new FileInputStream(path);
        byte[] data = fis.readAllBytes();
        return parseConfig(data);
    } catch (FileNotFoundException e) {
        System.err.println("文件不存在，使用默认配置");
        return DEFAULT_CONFIG;
    } catch (IOException e) {
        System.err.println("读取失败");
        return DEFAULT_CONFIG;
    } finally {
        if (fis != null) {
            try {
                fis.close();
            } catch (IOException e) {
                // 啥也不干
            }
        }
    }
}

    这段代码看起来很"健全"：try-catch 兜底了，finally 关了流。但它有几个问题：

    1. finally 里的 close 吞掉了原始异常 —— 如果 fis.close() 也抛异常，你永远不知道真正的问题是什么
    2. 代码臃肿 —— 关个流写了 10 行
    3. 容易忘记 close —— 今天写 FileInputStream 记得关了，明天写个 ZipInputStream 呢？

    你的 Java 程序，真的健壮吗？

    二、Java 异常世界观 —— Throwable 家族

    Java 的异常体系以 Throwable 为根，向下分为两大分支：

    Throwable（可抛出的）
├── Error（错误）—— 不可处理，程序应退出
│   ├── OutOfMemoryError        ← 内存溢出
│   ├── StackOverflowError      ← 栈溢出
│   └── ...
└── Exception（异常）—— 可处理
    ├── RuntimeException（运行时异常）—— 非受检
    │   ├── NullPointerException          ← 空指针
    │   ├── IllegalArgumentException       ← 非法参数
    │   ├── ClassCastException             ← 类型转换异常
    │   └── ...
    └── 其他 Exception（受检异常）
        ├── IOException                    ← IO 异常
        ├── SQLException                   ← SQL 异常
        └── ...

    三个家族的处理策略完全不同：

    类别 英文 处理要求 典型代表
    Error 错误 ❌ 不应捕获，程序应终止 OutOfMemoryError、StackOverflowError
    Checked Exception 受检异常 ✅ 必须处理（捕获或声明抛出） IOException、SQLException
    RuntimeException 运行时异常 ⚠️ 不强制处理，但建议处理 NullPointerException、IndexOutOfBoundsException

    核心原则：

    • Error —— 不要捕获。让程序崩溃，你去修代码。捕获 OutOfMemoryError 然后假装没事？不存在的。
    • RuntimeException —— 通常是程序 bug（空指针、越界），修复代码而非捕获异常
    • Checked Exception —— 可预见的异常（文件不存在、网络超时），必须处理

    三、受检 vs 非受检 —— 一场持续二十年的争论

    受检异常（Checked Exception）是 Java 独有的设计，C#、Python、Go 都没有。它强制开发者处理可预见的异常。

    // 受检异常 —— 编译不处理就报错
public void readFile(String path) throws IOException {
    // 编译器：喂！FileNotFoundException 和 IOException 你打算怎么办？
    FileInputStream fis = new FileInputStream(path);  // 可能抛 FileNotFoundException
    byte[] data = fis.readAllBytes();                 // 可能抛 IOException
}

// 方案一：try-catch 自己处理
public void readFile(String path) {
    try {
        FileInputStream fis = new FileInputStream(path);
        byte[] data = fis.readAllBytes();
    } catch (FileNotFoundException e) {
        System.err.println("文件不存在: " + path);
    } catch (IOException e) {
        System.err.println("读取失败: " + e.getMessage());
    }
}

// 方案二：throws 抛给上层
public void readFile(String path) throws IOException {
    FileInputStream fis = new FileInputStream(path);
    byte[] data = fis.readAllBytes();
}

    而非受检异常（RuntimeException 及其子类），编译器根本不 care：

    // 编译器不会强制你处理 —— 但运行时可能崩得很惨
String s = null;
s.length();  // ❌ 运行时报 NullPointerException，程序直接炸

    争议在哪里？

    受检异常是个有争议的设计。支持者说它让代码更健壮，反对者（包括很多大牛）说它导致代码臃肿。

    看看现代框架的态度：

    // Spring 的态度 —— 全是非受检
public class DataAccessException extends RuntimeException { }  // Spring 的数据访问异常
public class NoSuchBeanDefinitionException extends RuntimeException { }  // Spring 的 Bean 异常

// MyBatis 的态度 —— 全是非受检
public class PersistenceException extends RuntimeException { }

    业界趋势非常明显：受检异常在"失宠"。Spring、MyBatis、Hibernate 等主流框架都倾向于非受检异常。

    个人建议：项目中用非受检异常（继承 RuntimeException），只在"调用者必须处理否则会有严重后果"时才用受检异常。

    异常处理的 5 条黄金法则

    // ❌ 法则一违例：空 catch 吞没异常
try {
    // ...
} catch (Exception e) {
    // 啥也不做 —— 这是万恶之源！
}

// ❌ 法则二违例：用异常控制流程
try {
    Integer.parseInt(userInput);
} catch (NumberFormatException e) {
    // 用 if-else 不好吗？
}

// ✅ 正确做法
if (userInput != null && userInput.matches("\\d+")) {
    int val = Integer.parseInt(userInput);
}

// ✅ 法则三：捕获具体类型
try {
    fis = new FileInputStream(path);
    data = fis.readAllBytes();
} catch (FileNotFoundException e) {
    log.error("文件不存在，使用默认配置", e);
} catch (IOException e) {
    log.error("读取文件失败", e);
}

// ✅ 法则五：记录完整堆栈（不是只记 message！）
log.error("操作失败", e);                    // ✅ 记录完整堆栈
// log.error("操作失败: " + e.getMessage()); // ❌ 丢了堆栈，等于没记

    黄金法则速记：不吞没、不用异常控流程、捕获具体、尽早抛出延迟捕获、记完整堆栈。

    四、try-with-resources —— Java 最好用的语法糖之一

    4.1 传统 finally 的三宗罪

    回到开头的代码，try-finally 释放资源有三宗罪：

    罪一：嵌套噩梦

    // 关两个流，代码直接丑哭
FileInputStream fis = null;
BufferedReader reader = null;
try {
    fis = new FileInputStream("test.txt");
    reader = new BufferedReader(new InputStreamReader(fis));
    // ...
} finally {
    if (reader != null) {
        try { reader.close(); } catch (IOException e) { /* 又得处理 */ }
    }
    if (fis != null) {
        try { fis.close(); } catch (IOException e) { /* 又一次 */ }
    }
}

    罪二：异常覆盖

    try {
    throw new RuntimeException("原始异常");
} finally {
    throw new RuntimeException("finally 的异常");  // 覆盖了原始异常！
}
// 调用者只知道 "finally 的异常"，原始异常彻底丢了

    罪三：容易忘

    不是每个开发者都记得在 finally 里关资源。写业务逻辑的时候，一个分支忘了 close，线上就是文件句柄泄漏。

    4.2 try-with-resources 的优雅

    Java 7 引入了 try-with-resources，彻底解决了这些问题：

    // 一行声明，自动关闭，干干净净
try (FileInputStream fis = new FileInputStream("test.txt");
     BufferedReader reader = new BufferedReader(new InputStreamReader(fis))) {

    String line = reader.readLine();
    System.out.println(line);

} catch (IOException e) {
    log.error("读取文件失败", e);
}
// fis 和 reader 自动 close()，顺序与创建顺序相反

    解决了三宗罪：

    • ✅ 不用嵌套了 —— 括号里声明就行
    • ✅ 异常不会丢失 —— close 抛的异常被抑制（suppressed），原始异常保留
    • ✅ 不会忘 —— 编译器替你想了
    // 抑制异常机制演示
try (CustomResource res = new CustomResource()) {
    throw new RuntimeException("原始异常");
} catch (RuntimeException e) {
    System.out.println(e.getMessage());           // "原始异常"
    Throwable[] suppressed = e.getSuppressed();   // close 抛的异常在这里
}

    Java 9 还做了增强，允许使用 effectively final 的外部变量：

    // Java 9+
FileInputStream fis = new FileInputStream("test.txt");
BufferedReader reader = new BufferedReader(new InputStreamReader(fis));
try (fis; reader) {  // 直接在 try 中使用变量名
    String line = reader.readLine();
}

    4.3 对比 Python 的上下文管理器

    如果你写过 Python，会发现 try-with-resources 和 Python 的 with 语句思路如出一辙。

    Python 的 with 语句：

    # Python: with 语句自动管理资源
with open("test.txt", "r") as f:
    content = f.read()
# 文件自动关闭，无需显式 close()

    Python 通过上下文管理器协议（__enter__ / __exit__）实现，和 Java 的 AutoCloseable 接口异曲同工：

    # Python 自定义上下文管理器
class ManagedResource:
    def __enter__(self):
        print("获取资源")
        return self

    def __exit__(self, exc_type, exc_val, exc_tb):
        print("释放资源")
        # 返回 True 可以抑制异常
        return False

with ManagedResource() as res:
    print("使用资源")

    对比一下：

    对比维度 Java try-with-resources Python with 语句
    接口/协议 AutoCloseable.close() __enter__ / __exit__
    异常处理 抑制异常（getSuppressed） __exit__ 接收异常参数，返回 True 则抑制
    多个资源 分号分隔 try (A a; B b) 逗号分隔 with A() as a, B() as b
    引入版本 Java 7 (2011) Python 2.5 (2006)
    外部变量 Java 9+ 支持 effectively final 始终支持（contextlib.closing）

    Python 还有个很贴心的 contextlib 工具库：

    from contextlib import closing, contextmanager

# 方式一：closing 自动调用 close()
with closing(open("test.txt")) as f:
    content = f.read()

# 方式二：@contextmanager 装饰器（yield 前后 = __enter__ / __exit__）
@contextmanager
def managed_resource():
    print("获取资源")
    yield
    print("释放资源")

with managed_resource():
    print("使用资源")

    核心思想是相通的：把资源管理的样板代码交给语言/库，开发者只关注业务逻辑。

    有趣的是，Java 的 try-with-resources 和 Python 的 with 语句都诞生于 2000 年代中期到 2010 年代初这个"资源管理觉醒期"。C# 的 using 语句（2005）也走了同样的路。

    五、自定义异常的艺术

    当 JDK 提供的异常类型不够表达业务语义时，就需要自定义异常。

    5.1 选 Exception 还是 RuntimeException？

    这是自定义异常的第一个选择：

    // 选择一：受检异常（extends Exception）
public class BusinessException extends Exception {
    // 调用者必须 try-catch 或 throws
}

// 选择二：非受检异常（extends RuntimeException）
public class ServiceException extends RuntimeException {
    // 调用者按需处理
}

    怎么选？

    • 如果真的希望调用者必须处理这个异常（比如转账失败需要回滚），选 Exception
    • 大部分业务场景 —— 选 RuntimeException。Spring 事务回滚也只对 RuntimeException 生效

    5.2 好的异常设计长什么样

    // ✅ 推荐：统一异常类 + 错误码
public class BizException extends RuntimeException {

    private final int code;
    private final String msg;

    public BizException(int code, String msg) {
        super(msg);
        this.code = code;
        this.msg = msg;
    }

    public BizException(int code, String msg, Throwable cause) {
        super(msg, cause);  // 保留原始异常
        this.code = code;
        this.msg = msg;
    }

    public int getCode() { return code; }
    public String getMsg() { return msg; }
}

// 错误码用枚举，而不是散落在各处的魔法数字
public enum ErrorCode {
    USER_NOT_FOUND(1001, "用户不存在"),
    ORDER_NOT_FOUND(1002, "订单不存在"),
    INSUFFICIENT_BALANCE(1003, "余额不足"),
    EXCEED_LIMIT(1004, "超过限额");

    private final int code;
    private final String msg;
    ErrorCode(int code, String msg) {
        this.code = code;
        this.msg = msg;
    }
    public int getCode() { return code; }
    public String getMsg() { return msg; }
}

// 使用
throw new BizException(ErrorCode.USER_NOT_FOUND.getCode(),
                       ErrorCode.USER_NOT_FOUND.getMsg());

    5.3 不推荐的做法

    // ❌ 类爆炸：每个错误一个异常类
public class UserNotFoundException extends RuntimeException { }
public class OrderNotFoundException extends RuntimeException { }
public class ProductNotFoundException extends RuntimeException { }
// ... 定义了几十个只改了个名字的异常类

// ✅ 一个 BizException + ErrorCode 枚举就够了
// 别让你的异常类比业务类还多

    5.4 自定义异常设计清单

    原则 说明
    继承选择 希望调用者必须处理 → Exception；希望调用者按需处理 → RuntimeException
    包含异常码 数字或枚举，方便前端和调用方识别错误类型
    包含原始异常 构造函数接收 Throwable cause，不丢失原始堆栈
    避免类爆炸 统一异常类 + 错误码枚举，而不是每个错误一个类
    不可变设计 异常对象创建后不应修改，不提供 setter

    六、总结

    一句话总结本文

    受检异常处理不可恢复的问题，非受检异常暴露编程错误，try-with-resources 替你管理资源，自定义异常不要搞类爆炸。

    最佳实践清单

    1. Error 不要捕获，让程序崩溃
    2. RuntimeException 修复代码而非捕获
    3. 受检异常 只在"不处理就会有严重后果"时使用
    4. try-with-resources 代替 try-finally 释放资源
    5. 自定义异常 用统一异常类 + 错误码枚举
    6. 日志 记录完整堆栈：log.error("msg", e)
    7. 不要 用异常控制正常流程 —— if-else 更合适
