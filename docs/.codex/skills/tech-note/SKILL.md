# 技术笔记撰写 skill

你正在为用户的个人技术知识库撰写笔记。默认目标是发布到 **MkDocs Material** 主题的静态博客站，因此语法约定对齐 MkDocs Material。如果用户的知识库明显不是 MkDocs（比如 Obsidian、VuePress），语法可以按需调整，但**下面所有关于"如何把概念讲清楚"的核心信条与结构规范不变**。

## 核心信条

> **概念是笔记的灵魂**。作者相信：技术学习的关键是打通基础概念——概念清晰了，后续学习就会轻松很多。所以每篇笔记的第一要务是**把概念讲透**，而不是堆命令、堆代码、堆 API。

如果你发现某个概念只是把官方文档换了个说法，那你没写好。真正好的概念讲解应该做到：
- **让读者知道"是什么"** —— 用一句话给出定义
- **让读者知道"为什么存在"** —— 它解决了什么问题？没有它会怎样？
- **让读者建立心智模型** —— 用比喻、类比、对比、场景，让抽象概念变得可以想象
- **必要时配图** —— 结构、流程、层次这类东西，文字讲不如图讲

## 参考样本（可选，按需查阅）

本 skill 目录下有一个 `references/` 文件夹，用来存放**范本笔记**——用户觉得写得好、想让 AI 对齐风格的成品笔记。

**使用规则**：

- 第一次为某个用户/知识库写笔记时，建议**先扫一眼 `references/` 里有什么**（`ls` 或列目录即可），大致知道能对齐哪种风格
- **不要一次性把所有范本都 Read 进来**——那会浪费大量 context。只在下面这些情况下选择性 Read：
    - 用户明确指了某个范本："照 `LLM.md` 那种感觉写"
    - 你对某个具体细节不确定（比如 admonition 语法、章节序号写法、代码块注释密度）——挑最相关的一份翻一下
    - 用户第一次用这个 skill、`references/` 里有内容、你完全不知道对方风格——Read 其中 1-2 份感受一下
- 如果 `references/` 是空的，说明用户还没放范本，直接按本文的硬性规范写即可

`references/` 里可以放任何 `.md` 文件，文件名建议能自解释（例如 `tutorial-style.md`、`interview-style.md`、`cheatsheet-style.md`），这样 AI 能根据文件名判断该看哪份。

## 工作流程

### Step 1 —— 收集必要信息

用 AskUserQuestion 一次性问清楚（如果用户消息里已经明确了，就跳过对应问题）：

1. **主题**：这篇笔记要讲什么？（例：`Python 装饰器`、`Redis 持久化机制`）
2. **目标文件路径**：保存到哪里？（例：`docs/python/02_basic/decorator.md`）
3. **信息来源**：
   - 用户提供草稿或大纲 → 优先使用，你负责结构化 + 润色 + 补充
   - 用户只给主题 → **允许联网**（WebSearch / WebFetch 官方文档），但优先查权威来源：官方文档、RFC、知名开源项目 README、Hugging Face 文档等，不要用中文水文 CSDN 类
   - 用户指定"二次整理"已有笔记 → Read 那些文件后重构
4. **体量预期**：单个知识点（<300 行）or 系统章节（多个知识点串联）？

### Step 2 —— 视体量决定是否先交大纲

- **单个知识点 / 短笔记**：直接写正文，跳过大纲环节。
- **系统章节 / 长笔记**：先给出目录树（一级 + 二级标题，每节一行说明要讲什么、是否配图），等用户 approve 再写正文。这一步避免你写完 3000 行发现整体结构不对。

大纲格式示例：

```
## 一、xxx 概述
  ### 1.1 是什么
  ### 1.2 为什么需要它（对比：没有它时怎么办）
  <!-- 建议配图：xxx 在系统中的位置示意图 -->

## 二、核心概念
  ### 2.1 概念 A
  ### 2.2 概念 B
  <!-- 建议配图：A 和 B 的关系图 -->
...
```

### Step 3 —— 按硬性规范生成正文

直接写入用户指定的路径。写完后简要报告：文件路径、大致行数、建议配图的位置数量。

## 硬性规范（Checklist，逐条对齐）

### 结构

- [ ] 一级标题用**中文序号**：`## 一、xxx` / `## 二、xxx`（不要用 `## 1.`、不要用 `# xxx`）
- [ ] 二级用阿拉伯数字：`### 1.1 xxx`、`### 1.2 xxx`
- [ ] 三级用 `#### 1. xxx` 或普通 `#### xxx`
- [ ] **核心概念、必知必会**的章节标题末尾加 `⭐⭐⭐`，重要但非关键的加 `⭐⭐`
- [ ] 大节之间用 `---` 分隔（视觉呼吸感）

### 开头

- [ ] 第一行不要用 `# xxx` 直接从 `## 一、xxx 概述` 开始（MkDocs 会用文件名生成标题），也可用 `## xxx 概述` 作为第一节
- [ ] 若有官方文档/RFC/论文，紧跟一个 blockquote 列 1-3 个链接：
  ```markdown
  > 官方文档：[xxx](url) | [yyy](url)
  ```

### 概念讲解（最重要）

- [ ] **每个概念先给一句定义**，格式："**xxx**（英文原名, 简称）是……"，粗体标术语
- [ ] **紧接着解释"为什么"**：解决了什么问题，或者对比一个更笨的替代方案
- [ ] 抽象概念（架构、层次、流程、数据流）**主动建议配图**——写一个 HTML 注释占位符（见下文）
- [ ] 概念说清楚了就停，不要为了凑长度硬加"踩坑"、"总结"，作者不需要模板化文章
- [ ] 允许口语化过渡词："一句话："、"说白了"、"其实"、"为什么这么设计" —— 但用得克制、恰到好处

### 图片占位符（关键规则）

**作者会用其他 AI 生图，你只负责说明"这里应该画什么"，并且预留好一个可以直接填 `src` 的 HTML 图片块**。

**必须严格按下面这个组合写**——配图建议注释 + 一个空 `src` 的居中图片块，两个东西挨在一起：

```markdown
<!-- 配图建议
主题：xxx（一句话说明这张图讲的技术主体）
概念：xxx（要传达的核心概念是什么，例：LLM 的三层结构 / TCP 三次握手时序）
类型：架构图 / 流程图 / 时序图 / 层次图 / 对比图 / 数据流图
建议路径：assets/imgs/{分类}/{topic}/{序号}_{描述}.png
-->
<p align='center'>
    <img src='' alt='xxx（简短描述，便于日后搜索）'>
</p>
```

`src` 一定要**留空**——作者生图后会直接把路径填进去，不需要改结构。`alt` 填一个简短的图片说明。

**什么时候一定要提配图建议**：
- 讲**结构 / 架构 / 分层**（例：Transformer Block、LLM 整体结构、OSI 分层）
- 讲**流程 / 步骤 / 时序**（例：三次握手、RLHF 三阶段、请求生命周期）
- 讲**多个组件的关系**（例：主从复制、生产消费、Agent 编排）
- 讲**对比**且用表格讲不清楚（例：不同注意力机制对比）

**什么时候不需要配图**：命令行操作、API 用法、简单代码示例、纯 Q&A 型知识点。

**规则**：宁可多提配图建议（用户可以选择性画），也不要漏掉能大幅提升理解的图。

### admonition / callout 语法

统一用 **MkDocs Material** 语法（不用 Obsidian `> [!tip]`）：

```markdown
!!! tip "标题（可省略）"
    这里是内容，缩进 4 个空格。

!!! warning "生产环境注意"
    ...

!!! note "参考资料"
    ...

??? note "折叠内容（点击展开）"
    ...
```

**Tab 组用 `=== "标签"`**：

```markdown
=== "Python"
    ```python
    print("hi")
```

=== "Java"
    ```java
    System.out.println("hi");
    ```
```

**用途约定**：
- `!!! tip` —— 使用建议、最佳实践
- `!!! warning` —— 踩坑提示、生产环境注意事项
- `!!! note` —— 参考资料、扩展阅读、补充说明
- `!!! info` —— 概念补充、原理解释（可选，note 也行）
- `!!! danger` —— 严重错误、安全隐患
- `??? note` —— 想收起来的长内容（大段代码、详细步骤、附加对比表）

### 表格

- [ ] 只要有 **≥3 项对比**，一律用表格，禁止写成 bullet 罗列
- [ ] 表头精简，第一列一般是"操作/维度/命令名"，后面是各方案的对应值
- [ ] 表格中的代码用反引号包裹：`` `map.getOrDefault(k, 0)` ``

示例：

```markdown
| 操作 | Java (HashMap) | Python (dict) |
|------|----------------|---------------|
| 查值(带默认) | `map.getOrDefault(k, 0)` | `d.get(k, 0)` |
| 检查 Key | `map.containsKey(k)` | `k in d` |
```

### 代码块

- [ ] 代码块必须指定语言（```python、```java、```bash、```sql）
- [ ] **关键逻辑行加中文注释**，让读者能沿注释理解
- [ ] 双语对照（Java + Python 或 JS + TS）时用**两个连续代码块**，各自独立，第二块开头用注释切换

### 中英文混排

- [ ] 术语首次出现给完整英文 + 括号中文：`Composition API（组合式 API）`
- [ ] 术语反复出现用反引号：`` `Composition API` ``
- [ ] 中文和英文/数字之间**加空格**：`大约 41%`、`Redis 7.2 版本`（不是`大约41%`）

### 语气

- [ ] 保留"交流感"：可以说"其实"、"说白了"、"为什么这么设计"、"一句话："、"踩坑"、"黑盒"这类口语化词
- [ ] 但**不要滥用**，一段话最多一个口语化词，避免油腻
- [ ] 用**第二人称隐含**（不用出现"你"，但让读者感觉在跟他讲）：`GPT-3 的做法是……`（而不是"你要知道，GPT-3 做的事情是……"）
- [ ] 避免正式论文腔：不要写"综上所述"、"由此可见"、"值得注意的是"

### 结尾

- [ ] **不加落款**（不要"整理日期：2026年"这种）
- [ ] 长文结尾可以选做一节 `## 附录` 放边角内容（可选）
- [ ] 短文自然收束即可，不硬凑总结

- 引用官方文档时，用 blockquote 或 `!!! note` 标出，附上链接

## 反例（不要这样写）

**❌ 概念定义空洞**：
> LangGraph 是一个用于构建 Agent 的库，它非常强大。

**✅ 概念定义讲清**：
> LangGraph is very low-level, and focused entirely on agent **orchestration**。
> LangChain 的 Chain 天然是顺序执行的（DAG），即使有分支链也难以处理复杂编排；Agent 虽能实现 ReAct 模式，但执行过程是黑盒。为解决这些局限，LangGraph 应运而生——它专注于 Agent 编排（orchestration），提供循环图、状态管理、人工介入等底层能力。

**❌ 模板化四段式**：
> Redis 是什么：xxx。为什么需要它：xxx。怎么用：xxx。踩坑：xxx。一句话总结：xxx。

**✅ 概念说清就停**：
> Redis 是一个开源的**内存型键值数据库**，属于 NoSQL 中的键值存储类型。数据存内存，读写极快（10万+ QPS）；同时支持 5 种数据结构、持久化和集群，比"简单缓存"能做的事多得多。

**❌ 图片直接嵌路径（AI 猜的路径大概率是错的）**：
> `<img src='./xxx.png'>`

**❌ 只写配图建议，不给可填 src 的图片块（作者补图时要手动敲 HTML）**：
> `<!-- 配图建议 主题：xxx ... -->`

**✅ 配图建议 + 预留 src 的居中图片块**：
> ```
> <!-- 配图建议
> 主题：Redis 主从复制
> 概念：主节点向多个从节点同步数据的流向
> 类型：数据流图
> 建议路径：assets/imgs/db/redis/01_主从复制.png
> -->
> <p align='center'>
>     <img src='' alt='Redis 主从复制数据流'>
> </p>
> ```

**❌ 论文腔**：
> 综上所述，通过对 HTTP 协议的分析，我们可以看到……

**✅ 直接说**：
> HTTP 是无状态的——服务端不记得你上一次请求做过什么。这一点是后面所有 Cookie、Session、Token 机制存在的根源。
