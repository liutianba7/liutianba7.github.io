### 工作流为什么选择 LangGraph？

1. 状态管理：LangGraph 提供 TypedDict 状态管理，节点间状态传递更清晰
2. 条件路由：支持复杂的条件分支，动态路由
3. 并发执行：多路检索节点可以自动并发执行
4. 可视化：可以打印工作流的 ASCII 图，便于调试
5. 循环支持：支持循环结构，便于实现迭代优化

### 重排序后的文档怎么处理的？

当候选文档通过 Rerank 模型进一步得到分数后，我们会按照分数大小从大到小排序，然后采用`断崖检测截断`动态的确定当前要将给 LLM 前几个文档。这样做可以避免低分文档的混入。


### 你们系统是如何进行分片的？⭐⭐⭐

我们系统采用「标题粗切 + 大块递归细切 + 小块合并] 的策略，核心目标是保证尽可能保证每个 chunk 有完整的语义边界，同时大小均匀。

1）标题粗切：按 Markdown 的 #{1,6} 标题做分割，逐行扫描，遇到标题就新建一个 section。这里做了个代码块保护，遇到 时标记进入代码块，内部跳过标题识别，避免把代码里的 # 注释 误当成标题。如果文档完全没有标题，整个文档作为一个 chunk 兜底。

2）递归细切：粗切后超过 2000 字符的 chunk，用 LangChain 的 RecursiveCharacterTextSplitter 递归切割。分割符按优先级依次尝试：段落 → 换行 → 句号 → 逗号，优先在语义边界处切。设置 chunk_size=2000、chunk_overlap=400，20% 的重叠保证相邻 chunk 上下文不丢失。

3）小块合并：小于 500 字符的碎片，如果和前一个 chunk 属于同一个父标题，就合并到一起。

最终每个 chunk 带上 content、title、parent_title、part 这些字段，进入下游向量化。

### chunk_size = 2000 是一定切分成2000吗?

不是的，这只是递归分割器的兜底处理，也就是只有遍历完所有分隔符，还>2000，才会触发强制截断，其他时候都是按照设定好的分隔符切分，比如当前chunk是3000，按照段落切分以后，长度为1300，就不会继续切分了。
### overlap有什么用？为什么 chunk_overlap 是 400？

overlap 的目的是**避免语义跨 chunk 时上下文断裂**。

我们测试过不同 overlap，太小容易截断实体定义，太大会增加重复召回，也会减小每个chunk包含的**信息密度**，因此选择约 20% 作为工程上的折中。如果业务变化，我们也会重新验证这个参数。

### 你们如何评估切片的好坏呢？⭐⭐⭐


对于切片的本事评估，我们做的比较简单，就是去看每个chunk的语义，上下文是不是完整的，有没有被截断，因为我们切片的时候采用的是【先基于标题粗切 + 对于大块递归的、尽可能按找自然边界（比如段落，换行符，句号）细切 + 小块合并），同时，我们设置了overlap，一定程度上避免了**在自然边界切分后丢失上下文**。

而chunk本身是为下游向量化和召回服务的，我们对 RAG 整体过程做了比较详细的评估，比如召回率，倒数排名融合，这些指标也反映了切片的好坏


### 你们系统 RAG 的查询流程是什么？⭐⭐⭐

我们系统的 RAG 查询流程主要分以下几步分五个阶段：

1）**用户查询重写**：LLM 从用户查询中提取商品名并重写查询，然后向量化到标准名库做混合检索比对。评分大于 0.85 直接确认，0.6 到 0.85 给候选让用户选，小于 0.6 就拒识。识别失败走 Early-Exit 直接返回，不浪费计算资源。

2）多路召回（并行）：确认实体后三路同时执行——常规向量检索用改写后的问题做 BGE-M3 向量化，按商品名过滤在 Milvus 中召回 Top 5；HyDE 先让 LLM 生成假设性回答，用它向量化去匹配"答案空间"提升召回率；网络搜索通过 MCP 调外部搜索引擎补充最新信息。

3）RRF 融合：只融合同源的两路向量结果，公式是 score = 1/(k + rank)，k 默认 60，让排名影响更温和，不会单路第一名直接主导。输出 Top 10。

4）Cross-Encoder 精排：把 RRF 结果和 Web 搜索合并，用 BGE-Reranker 模型把 query 和 document 拼接输入做精细打分。做了动态 TopK 截断，相邻分数差超过阈值就提前截断，避免低分文档混入。异常时返回原始顺序不中断流程。

5）LLM 生成：精排后的文档组装成参考上下文，填到 Prompt 模板调 LLM。支持 SSE 流式输出，逐 token 推送，前端可以实时打字机展示。


### 如何实现的 pdf -> md 呢？⭐⭐⭐


采用 MinerU 云端 API 方案，整个流程分三步：

1）提交上传：先把文件信息 POST 到 MinerU 的 /file-urls/batch 接口，API 返回 batch_id 和一个预签名的 OSS 上传 URL。然后 PUT 把 PDF 二进制数据上传到阿里云 OSS，这里有个坑——需要指定 Content-Type: application/pdf，否则 OSS 会拒绝。

2）轮询等待：上传完后 GET /extract-results/batch/{batch_id} 轮询解析状态，每 3 秒查一次，state 字段为 done 表示完成，返回 full_zip_url。超时设了 60 秒，500 错误自动重试不中断轮询。

3）下载解压：拿到 ZIP URL 后下载到本地，解压到 local_dir。遍历 .md 文件，优先匹配和 PDF 同名的 .md，找不到就取 full.md，兜底取第一个，最后重命名为标准名称返回路径。

### pdf 中的图片是如何处理的？⭐⭐⭐

1）扫描图片：解析 MD 文件，通过正则匹配找到 images/ 目录下被引用的图片，并提取每张图片的上下文。

2）生成描述：把图片 Base64 编码后传给视觉大模型，生成语义描述文本。

3）上传 MinIO：图片上传到 MinIO 对象存储，拿到可访问的网络 URL。

4）替换标签：用 \![描述](MinIO_URL) 替换原来的 \[图名](本地路径)，保存到新的 MD 文件中，这样图片信息就能参与后续的向量化检索了。

### 嵌入模型选择的是什么？和其他模型做过对比吗？

我们选择的嵌入模型是 `BGE-M3`，选择它的原因一是因为它在**中文场景下表现好**，二是因为它能**同时生成稀疏向量和稠密向量**，方便我们后续的混合检索。最后是因**为它是开源的，部署起来比较简单，开销也不大**。

我们也对比过一些其他模型，比如 BGE-large-zh，它的优点是中文语义理解能力强，部署也很简单，但是缺点是只能生成：**dense vector（稠密向量）**。还考虑过 `OpenAI embedding`，OpenAI embedding 效果也很稳定，但是模型不开源，只能调用 api，同时中文场景不是最优的；

### 为什么选 Milvus 而不是 FAISS / Chroma？

我们选择 Milvus 而不是 FAISS 或 Chroma，主要考虑的是`生产级知识库场景`。
 
第一、它的分片、副本机制使得它天生支持分布式，方便后续拓展。这点和 ES 很像。这是 FAISS，Chroma，Qdrant 不具备的。
 
第二，**它天然支持向量和关键字混合检索**、对于向量检索，它提供了多种 ANN 索引，对于稀疏向量，也提供了对应的索引类型。我们的方案中使用 BGE-M3 同时生成 dense 和 sparse embedding，就是要在 Milvus 中进行混合检索.

第三，**Milvus 还支持元数据过滤等**，也就是标量检索，在很多业务场景下，标量检索也很重要，比如元数据里由存入知识库的日期，我们就可以实现在特定日期的知识文档实现召回。

最后，得益于 **Milvus** 提供的 ANN 索引和算法，在做向量召回时，Milvus的性能很高，
### 为什么解析 pdf 用 mineru，对比过其他方案吗？⭐⭐

我们当时对比过其他方案，比如最轻量化的 `PyMuPDF`、它的优点是轻量级、解析速度快，但是对于图片，其次还有了解过 `Marker`，Marker解析效果很好，但是需要本地部署模型，维护成本较高。OCR方案虽然识别能力强，但是需要自己组合版面分析和结构恢复； 

最终选择 MinerU，是因为它可以直接输出 Markdown，同时保留标题、表格、公式和图片信息，方便后续 chunk、embedding 和向量检索。同时通过 API 调用降低了系统维护成本。

### 如何评估你们 Rag 的好坏呢？⭐⭐⭐

我们评估 RAG 效果主要分三个方面。

第一是检索效果，我们构建了内部的测试集（200条），也就是人工标注问题对应的相关 chunk，然后使用 **Recall@K**， **Precision@K** 和 **MRR** 评估召回效果。Recall@K 主要衡量问题相关 chunk 的召回覆盖率，MRR 衡量正确 chunk 的排序位置。**Precision@K** 主要衡量返回结果的准确性，也就是召回的 Top-K chunk 中有多少是真正相关的，避免召回过多无关内容。

第二是生成效果，我们主要关注 **Answer Correctness（模型最终回答是否正确解决了用户问题。）** 和 **Faithfulness**（**模型回答是否严格来源于检索到的 Context，而不是自己编造**），判断模型回答是否正确，以及是否严格基于检索上下文，避免产生幻觉。

第三是线上效果，通过用户满意度、人工评分以及追问率等真实反馈指标，评估 RAG 在实际业务场景中的可用性。


- **HyDE、多路召回、Hybrid Search → 提升 Recall**
- **Rerank → 提升 Precision、MRR/NDCG**

### 如果你们的召回率很高，但是最后生成的答案效果不好，什么原因，怎么解决？

如果 Recall 很高但是最终答案效果不好，我认为问题主要在 **Retriever** 之后的环节。

第一，**可能是 rerank 阶段的问题**，虽然相关 chunk 被召回，但是没有被正确排序，导致最终交给 LLM 的 context 质量不好。

第二，**可能是 context 组织问题**，例如**召回 chunk 数量过多**，存在**大量噪声**，**或者 chunk 本身切分不合理**，导致信息不完整。

第三，**可能是 Prompt 或 LLM 生成能力的问题**，比如模型没有严格基于 context 回答，产生幻觉。

所以排查时我会结合不同指标定位：如果 Recall 高但 MRR 低，优先优化 rerank；如果检索指标正常但 Faithfulness 低，说明是生成阶段问题；如果 chunk 本身不完整，则优化 chunk 策略。


### 你们实际上如何完成的评估？⭐⭐⭐

**我们并没有用 rages 框架，而是自己写了评估的逻辑。**

对于检索效果：
`
首先，我们构建了测试的向量库，然后手动构建了一个测试集，每条样本就是一个问题，和当前问题相关的chunk（人工标注的）
``` json
{
    "id": "test_001",

    "question": "阿莫西林这个药如何服用？",

    "relevant_chunks": [
        "chunk_102",
        "chunk_103"
    ],

    "reference_answer": "阿莫西林通常需要按照医生指导服用，成人常见剂量为每天三次，每次xxx。"
}
```

对于每条样本，我们会走一个完成的 Rag 流程，然后得到评估需要的数据，主要就是LLM 最后生成的答案、真正召回的 chunk id 和 chunk 内容，之后，就可以计算各种评估指标了！

### 怎么计算的Answer Correctness和 **Faithfulness**

对于**Answer Correctness**，由于我们测试的规模集比较小，所以我们使用的是人工打分，首先，对于测试集的问题，我们会给出参考答案，然后，我们会收集 LLM 基于参考资料的回答，然后人工的去给出分数（1——5分），当然，如果数据规模大的话，我们可以引入 LLM 作为评估者（但是需要注意LLM评分的可靠性）

对于 **Faithfulness**，我们主要关注模型回答是否严格基于检索到的 context，避免幻觉。具体做法是保存每个问题对应的 **retrieved chunks** 和**最终回答**，然后人工检查回答中的**关键事实**是否能够**在检索内容中找到依据**，然后给出分数（1——5）。由于测试规模较小，**我们同样采用人工评分**（最终分数在3.7左右）；在数据量较大时，**可以结合 RAGAS 或 LLM Judge** 自动评估。


### 你认为你的工作有哪些可以改进的地方呢？

1、现在的评测都是手动构造数据，评测。我认为完全可以建立线上反馈闭环，将用户反馈和失败案例自动沉淀到评测集，持续优化检索和生成效果。


2、我会考虑引入**动态的检索策略**。目前系统是**固定检索链路**，对于复杂问题，可以增加 **query decomposition** 或者 agentic retrieval（让 Agent 参与 RAG 的检索决策过程，而不是固定执行一条检索链路）。

### 什么是 query decomposition

query decomposition 就是把一个复杂的问题拆分为若干子问题，然后每个子问题分别去做召回，从而尽可能全面的召回相关文档。

比如：**“对比阿莫西林和头孢的区别，它们分别适用于哪些疾病？”**，传统 Rag 在面对这样复杂问题的时候，可能会丢失一些重要的资料，而通过 **query decomposition**，可以将这个将这个问题拆分为多个子 Query：

- 阿莫西林是什么？适应症是什么？
- 头孢是什么？适应症是什么？
- 阿莫西林和头孢有什么区别？
