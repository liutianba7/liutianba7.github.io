    第二段实习

    第一部分：项目背景（100%会问）

    简单介绍一下这段实习？⭐⭐⭐

    这段实习是在一家北京的 AI 初创的公司完成的，项目目标是利用多 Agent 自动完成复杂的生信分析流程，例如对于一个 RNA-seq 的任务，它会先去查阅资料和文献，然后 coder 执行代码，产出结果， Reporter 根据资料和coder的产出做出汇报。

    传统的单 Agent 很难完成这种长链路任务，所以我们基于 LangGraph 搭建了一套多智能体工作流，把整个流程拆成多个角色协作完成。

    我主要参与了两个模块：第一是多 Agent 工作流的搭建，包括多个节点的搭建，状态的设计，以及如何进行动态路由等。第二是多 Agent 的三级记忆体系设计。

    第二部分 Multi-Agent（核心）

    为什么不用一个 Agent？而是 Multi-Agent⭐⭐⭐

    一个 Agent 更适合完成简单问答或者短流程任务，但我们的生信分析具有明显的阶段性，例如需要先规划分析流程，再查阅文献和资料，随后编写并执行分析代码，最后生成实验报告。

    如果全部交给一个 Agent，它需要同时承担所有角色，Prompt 会越来越复杂，历史上下文不断累积，推理容易受到无关信息干扰，任务边界也不清晰。

    因此，我们采用了多 Agent 的协作模式，由不同 Agent 分别负责不同阶段的任务，再通过 planner-supervisor 统一协调整个执行流程。

    你说拆成多个 Agent，那和一个 Agent 调多个 Tool 有什么区别？

    两者核心区别在于 Tool 只是提供能力，而 Agent 本身具有独立的推理目标。

    一个 Search Tool 只是负责返回搜索结果，但是 Researcher Agent 不仅调用搜索工具，还需要判断：

    • 搜什么关键词
    • 哪些资料可信
    • 如何总结信息

    所以 Multi-Agent 的核心不是简单拆工具，而是把不同阶段的决策过程独立出来。

    介绍一下你们 Multi-Agent 的架构⭐⭐⭐⭐⭐

    整体采用 Coordinator + Planner/Supervisor + Worker 的架构。

    Coordinator 负责接收用户请求和初始化任务。

    Planner 根据用户目标生成初始任务计划。

    计划确定后，Supervisor 接管调度，根据当前 checklist 状态决定下一步调用哪个 Worker。

    Worker 包括 Researcher、Coder、Reporter。

    为什么选择 LangGraph？

    因为我们的整个 Workflow 不是一个简单的 DGA（有向无环图），相反，我们的 WorkFlow 是存在环的、是动态的。

    而 LangGraph 本身就是一个底层的编排框架。

    它不仅提供精细的 State 管理，比如我们的 checklist, Agent的输出，中间结果都可以共享到任意一个节点。

    还支持动态路由，我们的 Workflow 中只有一条静态边，其他的全部是是同 Command(goto='')来实现的动态跳转。

    其次，LangGraph 也支持循环结构，比如我们的 Coder 如果发现代码存在问题，就会修改代码，再去执行，这就是一个典型的循环结构。

    为什么不用普通 python 写流程呢？

    普通 Python 当然可以实现，但是没必要，LangGraph 已经通过 State + Node + Edge 的抽象，使 Agent Workflow 更容易搭建和维护了。

    第三部分 任务调度

    简单说说 Planner-Supervisor？⭐⭐⭐

    在任务开始阶段，planner-supervisor 会分析用户需求，将复杂任务拆解成多个可执行步骤，并为每个步骤分配对应的 Worker Agent，生成结构化 checklist。

    在执行阶段，它会持续读取 checklist 状态以及 Worker 返回结果，判断当前任务进度，并根据当前状态决定下一步执行哪个步骤，然后通过 LangGraph 的动态路由机制调用对应 Worker。

    同时，如果执行过程中发现原计划不合理，例如某个步骤无法完成或者出现新的信息，它可以对 checklist 进行 patch 式调整，而不是重新生成整个计划。

    什么把 Planner 和 Supervisor 合并？为什么不拆开？

    因为如果把他两合并的话，我们的 Agent 数量会减少，同时任务规划和执行控制高度相关，所以它两在一起也比较合理。

    那这样是不是 Planner 每次都重新规划？

    初始阶段会生成完整 checklist。后续执行阶段主要是基于已有 checklist 做状态推进和局部 patch。只有当执行结果表明当前计划不可执行时，才触发重新规划。

    为什么采用 patch，而不是重新生成 plan？

    因为长任务执行过程中，大部分步骤仍然有效，如果每次都重新生成完整计划，会导致已有进度丢失，也就是 replan 的代价是比较大的。

    而 Patch 可以保留已经完成的步骤，只调整 pending 或者还没开始的步骤，这样代价小，同时还能动态的调整计划。

    Supervisor 怎么知道下一步执行谁？

    它是根据 checklist 中当前 pending 的 PlanStep，通过 step 中保存的 agent_name 字段进行路由。

    为什么引入 checklist

    因为长任务最大的问题是不确定性。如果完全依赖 Supervisor 的隐式记忆去判断当前进度和下一步动作，容易出现重复执行、遗漏步骤以及无法判断任务完成条件的问题。

    所以我们引入了显式的 checklist，将任务计划和执行状态结构化表示。从而清晰的告诉了 Supervisor 当前任务的完整流程以及当前的执行进度，当然，这个 checklist 不是一成不变的，Supervisor 根据 worker 返回的结果，可能会对还在 pending 或者未开始的步骤做 patch，如果计划完全不能执行了，还会触发 replan。

    第四部分 记忆系统

    你们的三级记忆系统具体是哪三级？⭐⭐⭐

    按作用范围和持久化程度分三层，从下往上越来越持久：

    层级 载体 范围 生命周期
    第一层 工作记忆 LangGraph State + messages 一次 run 内 run 开始创建，run 结束释放
    第二层 会话上下文 context_envelope.json 同 session 多次 run 之间 session 内持续，跨 run 保持
    第三层 长期事实 TemporalFact (temporal_facts.jsonl) 跨 session、跨多次 run 持久化存储，长期有效

    第一层：工作记忆——一次 run 内

    所有 Agent 共享同一个 State，通过 messages 传递对话历史，通过 structured fields（checklist、evidence、handoff）传递关键结果。问题是 run 一结束这层就没了——LangGraph 的 State 对象被释放了。下次 run 是一个全新的任务，不是上次 run 的延续。

    第二层：会话上下文——同 session 多次 run 之间

    第一次 run 结束后，系统不会把 3 万 token 的 messages 全存下来，而是提炼出一个 context_envelope.json：包含最近 3 次 run 的摘要、最近几轮对话、产出了哪些文件。第二次 run 启动时直接加载这个信封，Agent 启动时就能看到历史概况。

    这一层本质是"压缩版的会话记忆"——不是所有细节，只是概要。

    第三层：长期事实——跨 session、跨多次 run

    这一层更进一步。不是"上次 run 说了什么"，而是"根据过去多次 run，系统知道了什么"。比如 run 1 做了 RNA-seq 差异分析，run 2 做了另一批数据的分析。第二层会说"上次 run 用了 DESeq2，产出了 volcano.png"；第三层会说"事实：DESeq2 适用于有生物学重复的 RNA-seq 差异表达分析，来源：run_001 的 researcher + coder 验证，状态：current"。

    第二层是"历史记录"，第三层是"提炼出来的知识"。

    为什么需要三级？不能只用一层吗？⭐⭐⭐

    逐层解决的问题不同：

    • 只用第一层（工作记忆）：每次 run 都从零开始，没有任何历史继承。Agent 重复做同样的事。
    • 只用到第二层（会话上下文）：只有概要，没有精炼的事实。下次 run 知道"上次做过什么"，但不知道"上次学到了什么"。而且跨 session 就丢失了。
    • 有了第三层（长期事实）：知识可以沉淀。不同 session 之间的经验可以共享。

    三级的设计逻辑：工作记忆负责当前执行，会话上下文负责概览，长期事实负责知识沉淀。

    任务完成后，事实是怎么提取并沉淀到第三层的？⭐⭐⭐⭐

    完整链路：工作记忆 → 提取候选 → 裁决 → 写入长期事实 → 更新会话上下文

    Step 1：从工作记忆提取候选（Candidate Extraction） Run 完成后，从三个来源收集候选记忆： 1. Run 输出结果（final_answer、final_response、artifact refs） 2. 文本记忆（被标记为 session_reusable 的 context store items） 3. 结构化事实（facts_current 中 status 非 failed 的 fact）

    每个候选包含 text、keywords、evidence_refs、confidence 等字段，写入 <run_root>/memory_candidates/candidates.jsonl。

    Step 2：裁决（Decide Policy） decide_candidate() 按以下逻辑链决策： 1. 空文本或无 evidence_refs → skip 2. 已存在相同 ID → accept 3. 文本完全重复 → skip 4. 低信号 artifact → archive 5. 来源质量低且看起来冲突 → needs_review（进人工队列） 6. 同 topic 有新版本 → supersede（旧事实标记 superseded，新事实写入） 7. 全新事实 → accept

    其中步骤 5-6 会触发 LLM 裁决（llm_adjudicator.py），判断新旧事实关系（supports / contradicts / updates / duplicate / unrelated），有预算控制（默认 8 次/run）和缓存。

    Step 3：写入长期事实 根据决策结果写入： - temporal_facts.jsonl（版本化事实，含 status/valid_from/valid_to/supersedes_id） - temporal_edges.jsonl（关系边：fact_from_run、fact_supersedes_fact、fact_has_keyword 等 7 种） - episodes.jsonl（运行记录） - memory_decisions.jsonl（决策记录） - 重建 current_projection.json + .md（当前状态快照）

    Step 4：更新会话上下文 同时更新 context_envelope.json，把这次 run 的摘要写进去，供下次 run 启动时加载。

    裁决机制具体怎么工作的？⭐⭐⭐⭐

    裁决在 promotion_policy.py 的 decide_candidate() 中，核心逻辑链：

    1. 基础过滤：空文本或无 evidence_refs → skip；candidate 状态为 failed/retracted 且同 topic 已有 current fact → invalidate
    2. 精确去重：candidate ID 已在某个 current fact 的 source_candidate_ids 中 → accept（已存在）
    3. 内容去重：normalized_text 和已有 current fact 完全一致 → skip（重复）
    4. 低信号过滤：artifact observation 且信息量低 → archive
    5. 冲突检测：source_quality < 80 且文本看起来和已有事实冲突 → needs_review（进人工队列）或触发 LLM 裁决
    6. 版本更新：同 topic 有来自不同 run 的 current fact → supersede
    7. 新增：否则 → accept

    LLM 裁决（llm_adjudicator.py）用单独的模型（qwen3.5-flash，可配置）判断新旧事实的关系，返回 recommended_decision。有预算控制（默认 8 次/run）和缓存（按 candidate+fact hash）。

    长期事实库是怎么设计的？⭐⭐⭐

    基于文件系统，非数据库：

    <temporal_root>/
  temporal_facts.jsonl      ← 版本化事实（append-only）
  temporal_edges.jsonl      ← 关系边（7种类型）
  episodes.jsonl            ← 运行记录
  memory_decisions.jsonl    ← 决策记录
  current_projection.json   ← 当前状态快照（JSON）
  current_projection.md     ← 当前状态快照（Markdown，给 LLM 读）

    FactRecord 关键字段： id, text, status（current/superseded/invalidated/archived）, validity（valid_from/valid_to）, topic_key, topic_signature, source_quality, evidence_refs（来源 run + step）, run_id, supersedes_fact_ids, conflict_group_id

    版本链机制： 事实一旦写入不可修改，新事实通过 supersedes_fact_ids 指向旧事实，旧事实 status 设为 superseded。evidence_refs 记录每条事实来自哪个 run、哪个 step，可追溯到原文。

    7 种边类型： fact_from_run、fact_from_candidate、fact_from_episode、fact_has_keyword、fact_evidenced_by_artifact、fact_supersedes_fact、fact_invalidates_fact

    新任务启动时怎么召回的？⭐⭐⭐⭐

    主入口在 temporal_context_selector.py 的 select_temporal_context()，四层召回：

    1. 词法匹配：从用户 query 提取关键词（context_query.py），去 temporal_facts 做 lexical matching，召回直接命中的事实。
    2. 语义匹配：query 做 embedding，到 embedding cache 做相似度搜索（semantic_embedding.py），召回语义相关但关键词没命中的。
    3. 图邻居扩展：对前两步命中的事实，沿 temporal_edges 做 graph neighbor expansion，拿到关联节点。例如命中 "TP53"，一跳扩展拿到关联的 "breast_cancer"、"apoptosis" 等。
    4. LLM 重排序（temporal_reranker.py）：LLM 判断哪些真正相关，过滤噪声。

    最终渲染成 TEMPORAL CONTEXT PACK，注入 Agent 启动上下文。同时有备用路径 retrieve_memory_evidence_pack()（structured_retrieval.py），在时态存储不可用时做 fallback。

    角色投影具体是怎么实现的？⭐⭐⭐⭐

    在 memory_evidence.py（graph 适配层）里，分别有三个构建函数： - build_planner_memory_evidence_pack()：给 Planner，侧重任务背景、可用方法、数据源 - build_worker_memory_evidence_pack()：给 Researcher/Coder，侧重技术参数、代码示例 - build_reporter_memory_evidence_pack()：给 Reporter，侧重关键结论、证据链

    底层调用同一套检索，外层做 role-specific 的过滤、去重、格式化，还会和已有的 EvidenceCard 去重。

    同时在 session_context_projection.py 层面，有 planner.md / coder.md / reporter.md 三个投影文件，包含 session_summary、recent_runs、artifacts、memory_highlights、temporal_recall 等区块，受 ContextBudgetPolicy（可配置 max_chars、max_runs 等）控制。

    第二层（会话上下文）和第三层（长期事实）的区别和配合？⭐⭐⭐

    第二层：context_envelope.json - 记录"run 的历史"：做了什么、产出了什么、和用户的对话摘要 - 格式是压缩过的自然语言摘要 - 供下次同 session 的 run 了解"之前做到哪了" - 跨 session 丢失

    第三层：TemporalFact - 记录"run 学到了什么"：精炼的事实，带版本、证据链、来源 - 格式是结构化的事实记录 - 供任何 session 的任何 run 复用知识 - 跨 session 保持

    配合方式：run 启动时，先加载第二层的 context_envelope 了解任务背景，再检索第三层拿到相关知识，两者合并后通过角色投影分发给不同 Agent。

    LangGraph 本身有记忆功能吗？和你们的有什么区别？⭐⭐⭐

    LangGraph 提供的是 Store 机制——通用键值持久化存储。但它只解决"数据存哪里"，不解决问题的生命周期。

    我们的三级记忆系统解决了整套流程： - 第一层（工作记忆）复用 LangGraph State - 第二层（会话上下文）和第三层（长期事实）是我们自己在文件系统上实现的 - 我们有完整的提取→裁决→存储→召回→投影闭环，LangGraph Store 没有

    这个记忆系统解决了什么实际问题？⭐⭐⭐

    1. 跨任务知识复用：Researcher 在任务 A 查过 "TP53"，任务 B 遇到同样基因时通过第三层直接拿到，不需重新检索。
    2. 减少重复劳动：Coder 在任务 A 写过 RNA-seq 差异分析代码，新任务通过角色投影拿到示例直接修改复用。
    3. 可追溯审计：每条事实带 evidence_refs + supersedes_id，可追溯到原文和决策过程。
    4. 对抗幻觉：Agent 有历史事实做参考，不纯粹依赖模型内部知识。
    5. 关系推理：图一跳扩展发现间接关联，比如用户问 "TP53 在乳腺癌中的作用"，能带出乳腺癌的分析方法和信号通路。

    你们怎么评估记忆系统的效果？⭐⭐

    1. 召回精度：对 query 标注预期事实，看 lexical + semantic + graph neighbor 三层召回是否覆盖
    2. 任务效率：A/B 测试——开启 vs 关闭记忆系统，对比 Researcher 检索调用次数、Coder 修改轮次
    3. 事实准确率：抽样 facts_current.json 中的事实，对照 evidence_refs 追溯原文验证

    三级记忆和 LangGraph State 的关系是什么？⭐⭐

    LangGraph State 是第一层（工作记忆）。三级记忆系统在 State 之外独立构建了第二层和第三层：

    • 交互点 1：run 完成后从 State 提取候选事实，经裁决后沉淀到第二层和第三层
    • 交互点 2：新 run 启动时，从第二层加载历史摘要，从第三层召回相关知识，合并后注入 State 作为启动上下文
